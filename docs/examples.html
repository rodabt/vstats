Growth Metrics Deep Dive

SaaS Metrics Suite

Track key subscription business metrics: MRR, ARR, ARPA, ARPU, and retention rates.

Business Context: A B2B SaaS company needs to monitor revenue health, customer acquisition efficiency, and retention trends for monthly executive reviews. 
import vstats.growth

fn main() {
    // Revenue by plan tier
    mrr_by_tier := [85000.0, 120000.0, 65000.0] // Starter, Pro, Enterprise
    mrr := growth.monthly_recurring_revenue(mrr_by_tier)
    arr := growth.annual_recurring_revenue(mrr)

    println("Revenue Metrics:")
    println("  MRR: \$${mrr:,.0f}")
    println("  ARR: \$${arr:,.0f}")

    // Per-customer metrics
    accounts := 1250
    users := 3400
    arpa := growth.arpa(mrr, accounts) // Average Revenue Per Account
    arpu := growth.arpu(mrr, users) // Average Revenue Per User
    println("\nPer-Customer Metrics:")
    println("  ARPA: \$${arpa:.2f}")
    println("  ARPU: \$${arpu:.2f}")

    // Retention metrics
    churned := 38
    total_customers := 1250
    churn := growth.churn_rate(churned, total_customers)
    retention := growth.retention_rate(churned, total_customers)
    println("\nRetention:")
    println("  Churn Rate: ${(churn * 100):.2f}%")
    println("  Retention Rate: ${(retention * 100):.2f}%")

    // Net Revenue Retention (NRR) - expansion minus churn
    mrr_start := 100000.0
    mrr_end := 115000.0
    churn_mrr := 8000.0
    expansion_mrr := 23000.0
    nrr := growth.net_revenue_retention(mrr_start, mrr_end, churn_mrr, expansion_mrr)
    grr := growth.gross_revenue_retention(mrr_start, churn_mrr)
    println("\nRevenue Retention:")
    println("  Net Revenue Retention: ${(nrr * 100):.1f}%")
    println("  Gross Revenue Retention: ${(grr * 100):.1f}%")

    // Unit economics
    acquisition_spend := 85000.0
    new_customers := 125
    cac := growth.cac(acquisition_spend, new_customers)
    avg_lifespan := 18.0 // months
    ltv := growth.ltv(mrr, users, avg_lifespan)
    ltv_cac_ratio := growth.ltv_cac_ratio(mrr, users, avg_lifespan, acquisition_spend, new_customers)

    println("\nUnit Economics:")
    println("  CAC: \$${cac:.2f}")
    println("  LTV: \$${ltv:.2f}")
    println("  LTV:CAC Ratio: ${ltv_cac_ratio:.2f}x")

    // Payback period
    payback := growth.payback_period(cac, arpu)
    println("  Payback Period: ${payback:.1f} months")

    // Magic number (growth efficiency)
    magic := growth.magic_number(180000.0, 0.75, 85000.0)
    println("\nGrowth Efficiency:")
    println("  Magic Number: ${magic:.2f}")
    if magic > 0.75 {
        println("  Status: Efficient growth ✓")
    } else if magic > 0.5 {
        println("  Status: Moderate efficiency")
    } else {
        println("  Status: Inefficient - burning too much for growth")
    }
}
Key Interpretations:
  • LTV:CAC Ratio: >3x is healthy, <1x means losing money on acquisition
  • NRR > 100%: Existing customers are expanding (negative churn)
  • Magic Number > 0.75: Efficient growth; < 0.5 means overspending

Cohort Retention Analysis

Track user retention over time to identify when and why users churn.

Business Context: Users from Q4 marketing campaigns are churning faster than previous cohorts. Need to identify the retention drop-off points and compare cohort performance. 
import vstats.growth

fn main() {
    // Monthly cohorts with retention data
    cohort_names := ["Jan", "Feb", "Mar", "Apr", "May", "Jun"]
    initial_sizes := [1000, 1100, 1200, 1150, 1300, 1400]

    // Retention by month (absolute users remaining)
    retention_data := [
        [1000, 780, 620, 510, 430, 380], // Jan cohort
        [1100, 850, 680, 560, 470, 0],   // Feb (incomplete)
        [1200, 920, 740, 610, 0, 0],     // Mar
        [1150, 880, 700, 0, 0, 0],       // Apr
        [1300, 1000, 0, 0, 0, 0],        // May
        [1400, 0, 0, 0, 0, 0],           // Jun (only month 0)
    ]

    mut ca := growth.create_cohort_analysis(cohort_names, initial_sizes, retention_data)

    // Print retention matrix
    println("Retention Matrix (%):")
    println("-".repeat(70))
    print("Cohort   ")
    for i in 0..6 {
        print("  Month ${i}  ")
    }
    println("")

    for i, cohort in ca.cohorts {
        print("${cohort.name:8} ")
        for j in 0..6 {
            if j < ca.retention_matrix[i].len {
                print("  ${(ca.retention_matrix[i][j] * 100):6.1f}%")
            } else {
                print("    -")
            }
        }
        println("")
    }

    // Average retention by month
    println("\nAverage Retention by Month:")
    for i, avg in ca.avg_retention {
        bar_len := int(avg * 40)
        bar := "*".repeat(bar_len)
        println("  Month ${i}: ${bar} ${(avg * 100):.1f}%")
    }

    // Compare specific cohorts
    println("\nCohort Comparison:")
    jan_ret, may_ret := ca.compare_cohorts("Jan", "May")
    println("  Jan cohort M1 retention: ${(jan_ret * 100):.1f}%")
    println("  May cohort M1 retention: ${(may_ret * 100):.1f}%")

    improvement := (may_ret - jan_ret) / jan_ret * 100
    if improvement > 0 {
        println("  Improvement: +${improvement:.1f}% ✓")
    } else {
        println("  Decline: ${improvement:.1f}%")
    }

    // Monthly churn rates
    println("\nChurn by Period:")
    churn_rates := ca.churn_by_period()
    for i, churn in churn_rates {
        println("  Month ${i} → ${i+1}: ${(churn * 100):.1f}% churn")
    }
}
Reading Retention Curves:
  • Linear drop: Consistent churn, predictable LTV
  • Steep early drop: Onboarding issues, wrong user fit
  • Flattening curve: Loyal core users, healthy product-market fit
  • Benchmark: Good SaaS has 80%+ month 1 retention, 60%+ by month 6

Funnel Analysis & Optimization

Analyze conversion funnels to identify the biggest opportunities for improvement.

Business Context: E-commerce site with 68% cart abandonment. Need to identify which funnel stage has the biggest drop-off and quantify the revenue impact. 
import vstats.growth

fn main() {
    // Full e-commerce funnel
    stage_names := ["Visitors", "Product View", "Add to Cart", "Begin Checkout", "Purchase"]
    stage_users := [150000, 45000, 18000, 9000, 4500]

    funnel := growth.create_funnel(stage_names, stage_users)

    println("Funnel Overview:")
    println("-".repeat(60))
    println("  Total Visitors: ${stage_users[0]:,}")
    println("  Total Purchases: ${stage_users[4]:,}")
    println("  Overall Conversion: ${(funnel.total_conversion * 100):.2f}%")
    println("  Overall Drop-off: ${(funnel.drop_off_rate * 100):.1f}%")

    // Stage-by-stage analysis
    println("\nStage-by-Stage Analysis:")
    println("-".repeat(60))

    conversions := funnel.get_conversions()
    for conv in conversions {
        println("\n${conv.from_name} → ${conv.to_name}:")
        println("  Conversion Rate: ${(conv.rate * 100):.1f}%")
        println("  Drop-off Rate: ${(conv.drop_off_rate * 100):.1f}%")
        println("  Users Lost: ${(conv.from_users - conv.to_users):,}")
    }

    // Identify biggest opportunity
    worst := funnel.highest_drop_off()
    println("\n\nBiggest Opportunity:")
    println("  Stage: ${worst.from_name} → ${worst.to_name}")
    println("  Drop-off: ${(worst.drop_off_rate * 100):.1f}%")
    println("  Users Lost: ${(worst.from_users - worst.to_users):,}")

    // Revenue impact calculation
    avg_order_value := 85.0
    potential_recovery := f64(worst.from_users - worst.to_users) * 0.1 * avg_order_value // 10% improvement
    println("  Potential Revenue (10% fix): \$${potential_recovery:,.0f}/month")

    // Segment comparison
    println("\n\nSegment Comparison:")
    println("-".repeat(60))

    platform_data := {
        "Desktop": [100000, 35000, 14000, 7000, 3500],
        "Mobile": [50000, 10000, 4000, 2000, 1000]
    }

    funnels := growth.segment_funnel(platform_data)

    for platform, f in funnels {
        println("\n${platform}:")
        println("  Conversion: ${(f.total_conversion * 100):.2f}%")

        convs := f.get_conversions()
        for c in convs {
            if c.drop_off_rate > 0.6 {
                println("  ⚠ ${c.from_name} → ${c.to_name}: ${(c.drop_off_rate * 100):.0f}% drop-off")
            }
        }
    }
}
Prioritizing Fixes:
  • High volume + high drop-off: Best ROI for optimization
  • Segment differences: May indicate UX issues on specific platforms
  • Benchmark: Good e-commerce has 2-3% overall conversion, >60% cart-to-purchase

Marketing Attribution Analysis

Compare attribution models to understand true channel ROI and optimize budget allocation.

Business Context: Marketing team has \$10M budget across 6 channels. Last-touch attribution shows poor ROI for awareness channels. Need multi-model analysis for better allocation. 
import vstats.growth

fn main() {
    // Customer journeys (touchpoint sequences)
    journeys := [
        ["Facebook", "Google", "Email", "Direct"],
        ["Google", "Google", "Referral"],
        ["Facebook", "Instagram", "Google"],
        ["Organic", "Email", "Direct"],
        ["Google", "Facebook", "Email"],
        ["Referral", "Direct"],
        ["Instagram", "Facebook", "Google", "Email"],
        ["Google", "Google", "Direct"],
    ]

    conversions := [true, true, true, true, true, true, true, false]
    revenue := [150.0, 200.0, 180.0, 120.0, 175.0, 90.0, 220.0, 0.0]

    channel_costs := {
        "Facebook": 5000.0,
        "Google": 8000.0,
        "Instagram": 3000.0,
        "Email": 1000.0,
        "Organic": 500.0,
        "Referral": 2000.0,
        "Direct": 0.0
    }

    // 1. First-Touch Attribution
    println("1. First-Touch Attribution (awareness focus):")
    println("-".repeat(50))
    first_touches := ["Facebook", "Google", "Facebook", "Organic", "Google", "Referral", "Instagram", "Google"]
    first_touch := growth.first_touch_attributes(first_touches, conversions, revenue)
    mut first_total := 0.0
    for r in first_touch {
        first_total += r.revenue
        println("  ${r.channel:12} \$${r.revenue:,.0f} (${(r.attribution_score * 100):.1f}%)")
    }
    println("  Total: \$${first_total:,.0f}")

    // 2. Last-Touch Attribution
    println("\n2. Last-Touch Attribution (conversion focus):")
    println("-".repeat(50))
    last_touches := ["Direct", "Referral", "Google", "Direct", "Email", "Direct", "Email", "Direct"]
    last_touch := growth.last_touch_attributes(last_touches, conversions, revenue)
    mut last_total := 0.0
    for r in last_touch {
        last_total += r.revenue
        println("  ${r.channel:12} \$${r.revenue:,.0f} (${(r.attribution_score * 100):.1f}%)")
    }
    println("  Total: \$${last_total:,.0f}")

    // 3. Linear Attribution
    println("\n3. Linear Attribution (equal credit):")
    println("-".repeat(50))
    linear := growth.linear_attributes(journeys, conversions, revenue)
    mut linear_total := 0.0
    for r in linear {
        linear_total += r.revenue
        println("  ${r.channel:12} \$${r.revenue:,.0f} (${(r.attribution_score * 100):.1f}%)")
    }
    println("  Total: \$${linear_total:,.0f}")

    // 4. Time Decay Attribution
    println("\n4. Time Decay Attribution (recent touchpoints weighted):")
    println("-".repeat(50))
    touchpoint_days := [
        [14, 7, 3, 1],
        [30, 7, 1],
        [21, 14, 1],
        [7, 3, 1],
        [14, 3, 1],
        [1, 1],
        [21, 14, 7, 1],
        [7, 3, 1],
    ]
    time_decay := growth.time_decay_attributes(journeys, touchpoint_days, conversions, revenue, 7.0)
    mut decay_total := 0.0
    for r in time_decay {
        decay_total += r.revenue
        println("  ${r.channel:12} \$${r.revenue:,.0f} (${(r.attribution_score * 100):.1f}%)")
    }
    println("  Total: \$${decay_total:,.0f}")

    // 5. ROI Analysis
    println("\n5. Channel ROI (using Linear Attribution):")
    println("-".repeat(50))
    roi := growth.channel_roi(linear, channel_costs)

    for channel in ["Facebook", "Google", "Instagram", "Email", "Organic", "Referral"] {
        cost := channel_costs[channel] or { 0.0 }
        roi_val := roi[channel] or { 0.0 }
        println("  ${channel:12}: Spend \$${cost:,.0f}, ROI ${(roi_val * 100):.1f}%")
    }

    // Blended ROAS
    total_revenue := linear_total
    total_spend := 19500.0
    blended_roas := growth.blended_roas(total_revenue, total_spend)
    println("\nBlended ROAS: ${blended_roas:.2f}x")
}
Choosing an Attribution Model:
  • First-touch: Best for awareness campaigns, brand marketing
  • Last-touch: Best for conversion optimization, direct response
  • Linear: Fair distribution, good baseline for comparison
  • Time-decay: Realistic for long sales cycles, recent touchpoints matter

2. Experimentation Workflows

End-to-end examples for designing, running, and interpreting A/B experiments.

Workflow 1: Conversion Rate Test

Business Context: Your landing page has a 5% signup rate. You want to test a redesigned hero section and need to detect at least a 1 percentage-point improvement (5% → 6%). You have 80% power budget and use a 5% significance level.

Step 1: Calculate sample size before starting

import vstats.experiment

fn main() {
    // How many users do we need per group?
    // baseline_rate=0.05, mde=+0.01 (5%->6%), alpha=0.05, power=0.80
    plan := experiment.sample_size_proportions(0.05, 0.01, 0.05, 0.80)
    println('Need ${plan.n_per_group} users per group (${plan.total_n} total)')
    println('Implied effect size (h): ${plan.effect_size:.3f}')
    // Output: Need ~8100 users per group (16200 total)
}

Step 2: Run the experiment, then analyze

    // After collecting data:
    // Control (original): 405 signups from 8100 visitors
    // Treatment (new design): 520 signups from 8100 visitors
    result := experiment.proportion_test(405, 8100, 520, 8100)

    println('Control rate:   ${(result.rate_a * 100):.2f}%')   // 5.00%
    println('Treatment rate: ${(result.rate_b * 100):.2f}%')   // 6.42%
    println('Absolute lift:  +${(result.diff * 100):.2f}pp')   // +1.42pp
    println('Relative lift:  +${(result.relative_lift * 100):.1f}%') // +28.4%
    println('p-value:        ${result.p_value:.4f}')
    println('95% CI on diff: [${(result.ci_lower*100):.2f}pp, ${(result.ci_upper*100):.2f}pp]')
    println('Significant:    ${result.significant}')
    // Interpretation: The new design increased signups by 1.42pp (28% relative lift).
    // This exceeds our MDE of 1pp and is statistically significant. Ship it.

Workflow 2: Continuous Metric Test (Revenue)

Business Context: Average revenue per user is $45 with a standard deviation of $20. You want to detect a $3 increase (6.7% lift). Calculate sample size, run the experiment, then analyze with a t-test. 
import vstats.experiment

fn main() {
    // Step 1: Sample size
    // baseline_mean=45.0, std=20.0, mde=$3, alpha=0.05, power=0.80
    plan := experiment.sample_size_means(45.0, 20.0, 3.0, 0.05, 0.80)
    println('Need ${plan.n_per_group} users per group')
    println("Cohen's d: ${plan.effect_size:.3f}")  // 0.15 (small effect)

    // Step 2: Analyze after collecting data
    // (In practice, load your data from a file or database)
    control   := [43.2, 45.1, 46.8, 44.3, 45.7, 43.9, 47.2, 44.8, 45.5, 46.1]
    treatment := [47.8, 49.2, 46.5, 50.1, 48.3, 47.9, 49.5, 48.7, 50.2, 47.4]

    result := experiment.abtest(control, treatment)
    println('Control mean:   \$${result.control_mean:.2f}')
    println('Treatment mean: \$${result.treatment_mean:.2f}')
    println('Lift:           \$${result.treatment_mean - result.control_mean:.2f} (${(result.relative_lift * 100):.1f}%)')
    println("Cohen's d:      ${result.effect_size:.3f}")
    println('p-value:        ${result.p_value:.4f}')
    println('95% CI: [\$${result.ci_lower:.2f}, \$${result.ci_upper:.2f}]')
    println('Significant:    ${result.significant}')
}

Workflow 3: Sequential Testing (Safe Early Stopping)

Business Context: You're running a high-traffic experiment where results come in fast. You want to check daily and stop early if a clear winner emerges — without inflating your false positive rate. 
import vstats.experiment

fn main() {
    cfg := experiment.SPRTConfig{ alpha: 0.05, beta: 0.20, mde: 0.01 }

    // Simulate daily checks with cumulative data
    // Day 1: 500 users per group
    r1 := experiment.sprt_test(25, 500, 28, 500, cfg)
    println('Day 1: LLR=${r1.log_likelihood_ratio:.3f}, decision=${r1.decision}')

    // Day 5: 2500 users per group
    r5 := experiment.sprt_test(125, 2500, 160, 2500, cfg)
    println('Day 5: LLR=${r5.log_likelihood_ratio:.3f}, decision=${r5.decision}')

    // Day 10: 5000 users per group — effect is clear
    r10 := experiment.sprt_test(250, 5000, 340, 5000, cfg)
    println('Day 10: LLR=${r10.log_likelihood_ratio:.3f}, decision=${r10.decision}')

    match r10.decision {
        .reject_null      { println('Significant — ship the treatment!') }
        .accept_null      { println('Futile — effect is smaller than your MDE.') }
        .continue_testing { println('Keep running.') }
    }
    // Note: boundaries are log((1-beta)/alpha) and log(beta/(1-alpha))
    println('Upper boundary: ${r10.upper_boundary:.3f}')  // ~2.77
    println('Lower boundary: ${r10.lower_boundary:.3f}')  // ~-1.56
}

Workflow 4: Bayesian A/B Test

Business Context: You have a smaller experiment (300 users per group) and want to make a decision even before reaching classical significance. The Bayesian approach gives you a direct probability that variant B is better. 
import vstats.experiment

fn main() {
    // 30 conversions / 300 users vs 48 conversions / 300 users
    result := experiment.bayesian_ab_test(30, 300, 48, 300)

    println('A conversion rate (posterior mean): ${(result.posterior_mean_a * 100):.1f}%')
    println('B conversion rate (posterior mean): ${(result.posterior_mean_b * 100):.1f}%')
    println('')
    println('P(B beats A):   ${(result.prob_b_beats_a * 100):.1f}%')
    println('Expected loss (ship B): ${result.expected_loss_b:.5f}')
    println('Expected loss (ship A): ${result.expected_loss_a:.5f}')
    println('')
    println('95% Credible Interval for A: [${result.ci_lower_a:.3f}, ${result.ci_upper_a:.3f}]')
    println('95% Credible Interval for B: [${result.ci_lower_b:.3f}, ${result.ci_upper_b:.3f}]')

    // Decision rule: ship B if expected_loss_b < threshold
    threshold := 0.001
    if result.expected_loss_b < threshold {
        println('\nDecision: Ship B (expected loss ${result.expected_loss_b:.5f} < ${threshold})')
    } else if result.expected_loss_a < threshold {
        println('\nDecision: Keep A (expected loss of shipping B is too high)')
    } else {
        println('\nDecision: Collect more data')
    }

    // With an informative prior (you know the baseline is around 10%):
    informed_cfg := experiment.BayesianConfig{ alpha_prior: 10.0, beta_prior: 90.0 }
    informed := experiment.bayesian_ab_test(30, 300, 48, 300, informed_cfg)
    println('\nWith informative prior: P(B beats A) = ${(informed.prob_b_beats_a * 100):.1f}%')
}

3. Industry Case Studies

Real-world examples with business context, methodology, and impact.

Case Study 1: SaaS Cohort-Based Retention Strategy

Scenario

B2B SaaS project management tool with $2M MRR experiencing 15% monthly churn, particularly high among trial-to-paid converts.

Business Problem

  • High churn among trial-to-paid converts
  • Q4 cohorts showing 40% higher churn than previous quarters
  • Reactive retention offers coming too late

Goal

Reduce churn by 20% within 6 months through proactive identification of at-risk customers.

Methodology

  1. Cohort analysis by acquisition month and plan tier
  2. Feature usage pattern analysis
  3. Predictive churn model with logistic regression
  4. Threshold optimization for intervention triggers
import vstats.{growth, ml, stats, utils}

fn main() {
	// STEP 1: Cohort Analysis
	println("=== Cohort Analysis ===")

	cohort_names := ["Aug", "Sep", "Oct", "Nov", "Dec", "Jan"]
	initial_sizes := [200, 220, 250, 300, 280, 260]

	retention_data := [
		[200, 150, 120, 95, 75, 60],  // Aug: normal churn
		[220, 165, 130, 100, 80, 65], // Sep: normal
		[250, 175, 130, 90, 60, 40],  // Oct: higher churn ⚠
		[300, 195, 125, 75, 45, 30],  // Nov: very high ⚠⚠
		[280, 170, 100, 55, 30, 0],   // Dec: high ⚠
		[260, 150, 85, 0, 0, 0],      // Jan: early signs
	]

	mut ca := growth.create_cohort_analysis(cohort_names, initial_sizes, retention_data)

	// Identify problematic cohorts
	println("Month 2 Retention by Cohort:")
	for i, cohort in ca.cohorts {
		ret := ca.retention_matrix[i][2] * 100
		status := if ret < 50 { "⚠ CRITICAL" } else if ret < 60 { "⚠ At Risk" } else { "✓ OK" }
		println("  ${cohort.name}: ${ret:.0f}% ${status}")
	}

	// STEP 2: Feature Analysis
	println("\n=== Feature Correlation with Churn ===")

	feature_names := ["login_frequency", "feature_adoption", "team_size", "support_tickets", "session_duration"]
	churn_correlations := [-0.65, -0.58, 0.42, 0.38, -0.35]

	for i, name in feature_names {
		corr := churn_correlations[i]
		impact := if corr < -0.5 { "STRONG PROTECTIVE" } else if corr < -0.3 { "MODERATE" } else { "WEAK" }
		println("  ${name}: ${corr:+.2f} (${impact})")
	}

	println("\nTop Predictive Features:")
	println("  1. Login frequency (r = -0.65): Strongest predictor")
	println("  2. Feature adoption (r = -0.58): Users who adopt features stay")
	println("  3. Team size (r = +0.42): Larger teams at higher risk")

	// STEP 3: Predictive Model
	println("\n=== Churn Prediction Model ===")

	// Features: [login_freq, feature_adoption, team_size, support_tickets, session_duration]
	user_features := [
		[15.0, 0.8, 3.0, 2.0, 45.0],  // User 1
		[8.0, 0.4, 8.0, 5.0, 25.0],   // User 2 (high risk)
		[20.0, 0.9, 2.0, 1.0, 60.0],  // User 3 (low risk)
		[5.0, 0.2, 10.0, 8.0, 15.0],  // User 4 (very high risk)
		[18.0, 0.85, 4.0, 2.0, 55.0], // User 5
	]
	churn_labels := [0, 1, 0, 1, 0] // 1 = churned

	// Split and train
	split_idx := 3
	x_train := user_features[0..split_idx]
	y_train := churn_labels[0..split_idx]
	x_test := user_features[split_idx..]

	model := ml.logistic_classifier(x_train, y_train.map(f64(it)), 1000, 0.01)
	predictions := ml.logistic_classifier_predict(model, x_test, 0.5)
	probas := ml.logistic_classifier_predict_proba(model, x_test)

	println("Model Performance:")
	println("  Tested on ${x_test.len} users")
	for i, pred in predictions {
		risk := if probas[i] > 0.7 { "HIGH" } else if probas[i] > 0.4 { "MEDIUM" } else { "LOW" }
		println("  User ${split_idx + i + 1}: ${probas[i] * 100:.0f}% churn probability (${risk} RISK)")
	}

	// STEP 4: Intervention Threshold Analysis
	println("\n=== Intervention Strategy ===")
	println("Threshold Analysis:")
	println("  0.7 threshold: High precision, fewer interventions")
	println("  0.5 threshold: Balanced precision/recall")
	println("  0.3 threshold: High recall, more false alarms")
	println("\nRecommended: 0.5 threshold with tiered response:")
	println("  > 70%: Immediate outreach + account review")
	println("  50-70%: Automated email sequence")
	println("  30-50%: Monitor closely")

5. Statistical Guidance

When to use which test, and how to interpret results correctly.

Choosing the Right Statistical Test

Decision Tree

What type of data do you have?

├── CONTINUOUS (revenue, time, ratings)
│   │
│   ├── One group?
│   │   └── One-sample t-test
│   │       "Is our NPS significantly > 50?"
│   │
│   ├── Two groups?
│   │   ├── Paired (same users before/after)
│   │   │   └── Paired t-test
│   │   │       "Did the treatment change user behavior?"
│   │   │
│   │   └── Independent (different users)
│   │       └── Two-sample t-test
│   │           "Is variant A different from variant B?"
│   │
│   └── 3+ groups?
│       └── ANOVA
│           "Do three pricing tiers have different conversion rates?"
│
├── CATEGORICAL (converted/didn't, device type)
│   │
│   ├── One variable?
│   │   └── Chi-squared goodness of fit
│   │       "Is our user base 50/50 mobile/desktop?"
│   │
│   └── Two variables?
│       └── Chi-squared test of independence
│           "Is device type associated with conversion?"
│
└── RANKINGS/SKEWED (revenue often skewed)
    │
    ├── Two groups?
    │   └── Mann-Whitney U test
    │       "Compare revenue between groups (non-normal)"
    │
    └── Paired?
        └── Wilcoxon signed-rank test
            "Before/after with non-normal distribution"

Test Comparison Table

Test Use When Example Question
One-sample t Compare group to known value Is our conversion rate > 3%?
Two-sample t Compare two independent groups Is variant A different from B?
Paired t Before/after same subjects Did users spend more time after feature?
ANOVA Compare 3+ groups Do three pricing tiers differ?
Chi-squared Categorical association Is device type related to conversion?
Mann-Whitney Non-normal distributions Compare revenue (skewed) between groups

Interpreting Results

p-values

✓ Correct Interpretation

"If there were no real effect, we'd see results this extreme by chance alone."

p < 0.05 means: "The data would be unlikely under the null hypothesis."

✗ Common Misinterpretation

"There's a 95% chance the effect is real."

"p < 0.05 means the effect is definitely real."

These are WRONG. p-value is about the data, not the hypothesis.

Effect Sizes

Why it matters: Statistical significance ≠ practical significance.

With enough data, tiny effects become "significant." Always report effect size.

Cohen's d Interpretation Business Impact
0.2 Small Noticeable but may not justify change
0.5 Medium Meaningful improvement
0.8 Large Clear business impact

Confidence Intervals

Interpretation: "We are 95% confident the true effect lies in [lower, upper]."

✓ Clear Result

CI: [+5%, +15%]

Doesn't include 0, effect is positive, range is narrow.

? Inconclusive

CI: [-2%, +18%]

Includes 0, could be positive or negative. Need more data.

✗ No Effect

CI: [-8%, +4%]

Centered near 0, effect is likely null or very small.

Common Misconceptions in A/B Testing

❌ "p < 0.05 means 95% chance the effect is real"

Reality: p-value doesn't tell you the probability the hypothesis is true. It tells you how surprising the data would be if there were no effect.

❌ "No significant difference means the variants are the same"

Reality: Absence of evidence ≠ evidence of absence. You might just need more data. Check the confidence interval.

❌ "We can stop the test once p < 0.05"

Reality: This is "peeking" and it inflates your false positive rate. Pre-register your sample size and stopping criteria.

❌ "More data always gives better results"

Reality: More data finds smaller effects, but tiny effects may not matter. Balance statistical power with practical significance.

6. Anti-Patterns & Pitfalls

Common mistakes and how to avoid them.

Anti-Pattern 1: Peeking at Results

Wrong: Checking p-value Daily

// WRONG: Stopping when p < 0.05
for day in 0..30 {
    result := experiment.abtest(control_so_far, treatment_so_far)
    if result.p_value < 0.05 {
        println("Significant! Stopping experiment.")
        break // This inflates false positive rate!
    }
    // collect more data...
}

Problem: With 10 peeks, your actual false positive rate is ~20%, not 5%.

Correct: Pre-register Sample Size

// RIGHT: Calculate sample size first
pa := experiment.sample_size_proportions(0.05, 0.01, 0.05, 0.8)
n_required := pa.n_per_group

// Wait until sample size reached before analyzing
if current_sample >= n_required {
    result := experiment.abtest(control, treatment)
    // Now interpret results
}

Solution: Pre-register your stopping criteria. Only analyze when you hit the target sample size.

Anti-Pattern 2: Data Leakage in ML

Wrong: Normalize Before Split

// WRONG: Uses test data statistics!
normalized := utils.normalize_features(all_data)
train := normalized[0..split_idx]
test := normalized[split_idx..]

// Problem: mean/std include test data
// This is "cheating" - model sees test info during training

Correct: Split First, Then Normalize

// RIGHT: Split first
train_data := all_data[0..split_idx]
test_data := all_data[split_idx..]

// Normalize using ONLY training statistics
train_norm, means, stds := utils.normalize_features(train_data)
test_norm := utils.apply_normalization(test_data, means, stds)

// Now train on train_norm, evaluate on test_norm

Anti-Pattern 3: Multiple Comparisons

Wrong: Testing Many Metrics Uncorrected

// WRONG: Testing 20 metrics
metrics := ["conversion", "revenue", "time_on_site", ...] // 20 metrics
for m in metrics {
    if m.p_value < 0.05 {
        println("Significant on ${m}!")
        // Expect ~1 false positive by chance!
    }
}

Problem: With 20 tests at alpha=0.05, expect 1 false positive even if all null hypotheses are true.

Correct: Apply Multiple Comparison Correction

// RIGHT: Bonferroni correction
alpha_corrected := 0.05 / metrics.len // 0.0025 for 20 metrics

for m in metrics {
    if m.p_value < alpha_corrected {
        println("Significant on ${m} after correction!")
    }
}

// Alternative: False Discovery Rate (FDR) control for less conservative

Anti-Pattern 4: Simpson's Paradox

Scenario

Overall: Treatment +5% conversion

Mobile users: Treatment -3% conversion

Desktop users: Treatment -2% conversion

Paradox: Treatment looks good overall but bad in every segment!

Explanation: Treatment attracted more mobile users (who have lower baseline conversion).

Correct: Always Check Segments

// Check segment-level effects
segments := ["mobile", "desktop", "tablet"]
for seg in segments {
    seg_control := filter(control, segment == seg)
    seg_treatment := filter(treatment, segment == seg)
    result := experiment.abtest(seg_control, seg_treatment)
    println("${seg}: ${(result.relative_lift * 100):.1f}%")
}

Anti-Pattern 5: Optimizing Wrong Metric

Wrong: Accuracy on Imbalanced Data

// WRONG: 95% accuracy sounds great!
// But if 95% of users don't convert,
// predicting "no conversion" for everyone gives 95% accuracy

baseline := 0.95 // 95% don't convert
dummy_accuracy := baseline // "Predict all negative" accuracy
println("Dummy model accuracy: ${dummy_accuracy * 100:.0f}%") // 95%!

Correct: Use Appropriate Metrics

// For imbalanced data, use:
// - Precision, Recall, F1
// - AUC-ROC
// - Precision-Recall AUC

cm := ml.confusion_matrix(y_test, predictions)
println("Precision: ${cm.precision():.3f}") // TP / (TP + FP)
println("Recall: ${cm.recall():.3f}") // TP / (TP + FN)
println("F1: ${cm.f1_score():.3f}") // Harmonic mean

// For ranking quality:
roc := utils.roc_curve(y_test, probas)
println("AUC: ${roc.auc:.3f}")

During Experiment

  • ☐ Don't peek at p-values
  • ☐ Check guardrail metrics

Post-Experiment

  • ☐ Run appropriate statistical test
  • ☐ Calculate confidence intervals
  • ☐ Report effect size (Cohen's d)
  • ☐ Check segment-level effects
  • ☐ Assess practical significance
