#The Counterfactual Question

Every experiment asks one question: what would have happened to these users if we had done nothing?

WHEN

Before designing or interpreting any experiment. Use this framing whenever a stakeholder asks "did the feature cause X?"

WHY

A causal effect is the difference between two potential outcomes: what happened under the treatment versus what would have happened to the same units under control. We can never observe both for the same user at the same time — this is called the fundamental problem of causal inference. Randomization solves it by making the control group a valid stand-in for the treatment group's counterfactual.

Without this framing, "the conversion rate went up after we shipped" is not evidence of causation — it could be seasonality, a simultaneous marketing push, or the fact that only high-intent users triggered the new feature.

HOW

  1. Write the counterfactual explicitly: "If user X had seen variant A instead of variant B, their 30-day revenue would have been…"
  2. Identify what outcome you are measuring and over what time window.
  3. Identify the unit of analysis (user, account, session).
  4. Ask: "Is the control group a plausible stand-in for what the treatment group would have done without the treatment?" If yes, the comparison is valid.

PITFALLS

  • Pre/post comparisons are not counterfactuals. Before-treatment performance of treated users is not what they would have done without treatment — time changes things.
  • Self-selected comparison groups are not counterfactuals. Users who discovered a new feature on their own differ systematically from those who didn't.
  • The counterfactual must be well-defined. "What if we hadn't had a pricing page?" is not answerable. "What if this user had seen the old pricing page instead of the new one?" is.

#Exchangeability, Positivity, Consistency

Three conditions that together make a causal estimate from data trustworthy.

WHEN

When auditing whether an experiment (or observational analysis) can support a causal claim. Run through these three checks before finalizing a readout.

WHY

These conditions come from the causal inference literature (Hernán & Robins) and map directly onto common experiment failures:

  • Exchangeability (no unmeasured confounding): treatment and control groups would have had the same outcomes had they both received the same treatment. Randomization buys this. Observational studies must argue for it.
  • Positivity (overlap): every unit has a non-zero probability of being in either group. Violated when, for example, enterprise accounts can only ever be in control, or when a feature is only triggered by a user action that correlates with the outcome.
  • Consistency (well-defined treatment): the treatment is the same for everyone labeled "treated." Violated when "saw the feature" means different things for different users (e.g., saw it once vs. used it 10 times).

HOW

  1. Exchangeability check: Run a balance table — compare pre-experiment covariate means across variants. Large imbalances suggest assignment is broken. Also check for SRM (see SRM card).
  2. Positivity check: Plot the overlap of key covariates (account age, plan, usage) across variants. Identify any covariate values with zero representation in one arm.
  3. Consistency check: Define "treated" precisely. Is it assignment to the variant, first exposure, or N exposures? Log the exact trigger and make sure your metric window starts from the same event.

PITFALLS

  • Partial exposure violates consistency. If users assigned to treatment never saw it (low trigger rate), your ITT estimate dilutes the true effect. See ITT vs per-protocol.
  • Positivity violations create extrapolation. When you adjust for covariates in groups where there is no overlap, the model is extrapolating, not estimating.
  • Exchangeability is never provable, only arguable. The strongest argument is a well-designed randomized experiment plus an SRM check.

#Reading a DAG in SaaS

A causal diagram (DAG) shows what causes what — and reveals whether a naïve comparison will give a biased answer.

WHEN

When you suspect a metric is confounded, when you're choosing which variables to control for in a regression, or when a stakeholder argues "but users who X also tend to Y" as an objection.

WHY

A Directed Acyclic Graph (DAG) encodes causal assumptions as arrows between variables. An arrow from A → B means "A causes B." A path between treatment T and outcome Y through a third variable C means C is a confounder if it has arrows into both T and Y. The key rule: controlling for a confounder removes bias; controlling for a collider introduces bias.

In SaaS, the two most common mistakes are: 1. Forgetting to account for a confounder (e.g., account size affects both who gets the feature and revenue) 2. Accidentally conditioning on a collider (e.g., filtering to "engaged users" when engagement is caused by both the treatment and baseline quality)

HOW

  1. List the treatment T, outcome Y, and all plausible third variables.
  2. Draw arrows based on domain knowledge (what causes what, not what correlates with what).
  3. Identify backdoor paths — non-causal paths from T to Y through shared common causes. These paths create confounding bias.
  4. Block backdoor paths by conditioning on (adjusting for) the confounders on those paths.
  5. Never condition on colliders — variables caused by two other variables — unless you account for the bias it introduces.

PITFALLS

  • Conditioning on a mediator (a variable on the causal path from T to Y) blocks the effect you're trying to measure. Don't control for engagement if engagement is how the feature works.
  • Collider bias is invisible unless you draw the DAG. A classic SaaS example: filtering to "converted users" when conversion is caused by both treatment and user quality creates a spurious negative correlation between treatment and quality.
  • DAGs encode assumptions, not facts. The value is in making your assumptions explicit and arguable.

#Association vs. Causation

A statistically significant result tells you the groups differed. It does not tell you the feature caused the difference.

WHEN

Any time a result will be used to justify shipping, killing, or modifying a feature — especially when a stakeholder says "we should ship this, it clearly worked."

WHY

Two variables are associated when knowing one tells you something about the other. They are causally related when intervening on one changes the other. In experiments, the randomization mechanism is what licenses the causal interpretation — it ensures the groups were exchangeable before treatment. Without randomization (or a valid quasi-experimental design), an association can be explained by confounders, selection effects, or reverse causation.

The practical test: would the effect replicate if you ran the experiment again on a fresh randomized cohort? If yes, and randomization was clean, you have a causal estimate. If the "experiment" was a comparison of users who self-selected into a feature vs. those who didn't, you have an association.

HOW

  1. Verify randomization was clean: check SRM, balance table, A/A test history.
  2. Identify any mechanism by which the result could be explained without the feature having an effect (novelty, self-selection, confounding by account size or plan).
  3. If randomization was clean and the mechanism check passes, the causal claim is warranted. State the scope: "among users eligible for the experiment, in the treatment window, on this metric."
  4. If randomization was imperfect or the study was observational, use one of the causal methods in 05-causal-methods.md and state the assumptions explicitly.

PITFALLS

  • "Correlation is not causation" is not a rebuttal. In a clean RCT, the correlation IS the causal estimate. The question is whether the RCT was clean.
  • Post-hoc segment comparisons are associations, not causal effects. Users in segment A vs. B were not randomized to their segment. See HTE.
  • Directionality: high engagement → high revenue can run either way. An experiment intervenes on one variable; any change in the other is causal in that direction.

#Randomization Unit

The unit you randomize determines the unit you analyze — and getting it wrong inflates false positives or dilutes real effects.

WHEN

Any time you're setting up an experiment. The default "randomize by user" is often wrong in B2B SaaS.

WHY

The randomization unit is the entity that gets assigned to a variant. Every analysis must aggregate to that same unit. If you randomize by account but analyze by user, you count some accounts more than others (large accounts dominate), and your standard errors are too narrow because users within the same account are correlated — this is the Intraclass Correlation (ICC) problem.

Choosing the wrong unit causes two distinct failures: 1. Contamination — treatment leaks from treated users to control users who share an account, workspace, or feed (see ICC card). 2. Wrong variance estimate — your t-test assumes independent observations; correlated observations within the same cluster produce a false sense of precision.

HOW

Scenario Randomize by
Change affects the whole account (billing page, permissions, dashboard layout) Account
Change is user-specific within an account (onboarding flow, email, recommendations) User
Anonymous or pre-login flow (landing page, signup funnel) Session / cookie
Supply-side or marketplace effects (driver incentives, listing ranking) City / region
Feature where users share a workspace and can see each other's actions Account or cluster

Steps: 1. Ask: "Can a user in control and a user in treatment interact in a way that affects the outcome?" 2. If yes → randomize by the larger unit that contains the interaction (account, workspace, city). 3. Recalculate power using the design effect DEFF = 1 + (m̄ - 1) × ICC, where m̄ is average cluster size. Your required sample size multiplies by DEFF. 4. Document the randomization unit in the experiment spec before launch.

PITFALLS

  • Randomizing by user for account-level features. Two users in the same account will see different variants — UX chaos and contamination.
  • Analyzing by user when you randomized by account. Your n looks large but your effective n is the number of accounts.
  • Mixing randomization and analysis units. The SQL GROUP BY must match the randomization unit.
  • Session randomization for logged-in flows. The same user gets re-randomized every session — violates stable unit treatment value (SUTVA).

import numpy as np
import matplotlib.pyplot as plt
import matplotlib as mpl
from scipy import stats

mpl.rcParams.update({
    'axes.spines.top': False, 'axes.spines.right': False,
    'axes.grid': False, 'font.family': 'sans-serif',
    'text.color': '#333', 'axes.labelcolor': '#333',
    'xtick.color': '#333', 'ytick.color': '#333',
})

rng = np.random.default_rng(42)

# Simulate: 200 accounts, avg 5 users each, ICC = 0.15
n_accounts   = 200
users_per    = 5
icc          = 0.15
account_fx   = rng.normal(0, np.sqrt(icc), n_accounts)
user_noise   = rng.normal(0, np.sqrt(1 - icc), (n_accounts, users_per))
y            = (account_fx[:, None] + user_noise).flatten()

# True treatment effect = 0.2 SD
true_effect = 0.2
variant      = np.repeat(rng.choice([0, 1], size=n_accounts), users_per)
y_obs        = y + true_effect * variant

# Wrong: t-test on user level (ignores clustering)
ctrl_u = y_obs[variant == 0]
trt_u  = y_obs[variant == 1]
t_wrong, p_wrong = stats.ttest_ind(trt_u, ctrl_u)

#Baseline and Time Zero

Baseline is what you compare against. Time zero is when the experiment starts. Getting both wrong is how you manufacture fake results.

WHEN

During experiment design (defining the baseline metric window) and during analysis (defining the outcome measurement window).

WHY

Time zero is the moment a unit enters the experiment — when they were assigned, not when they first triggered the feature. If you define time zero as first exposure, you introduce immortal time bias: units that never trigger the feature (and may churn or be low-intent) are excluded, inflating your treatment effect.

Baseline period is the pre-experiment window used to measure the metric's historical level for CUPED, balance checks, and sanity checks. It must: - Precede assignment (not overlap with treatment) - Be long enough to capture seasonality (≥ 7 days, ideally 14) - Use the same metric definition as the outcome

HOW

  1. Set time zero = assignment date. Record the UTC timestamp when each unit was randomized.
  2. Define the outcome window starting from time zero (e.g., D0–D30).
  3. Define the baseline window ending before time zero (e.g., D-14 to D-1).
  4. Verify no treatment events fall in the baseline window — if a feature was soft-launched before formal assignment, the baseline is contaminated.
  5. For CUPED, use the baseline metric as the covariate (see CUPED card).

PITFALLS

  • Immortal time bias: defining entry as "first feature use" excludes users who never use it. Include all assigned users, revenue = 0 for non-users (ITT).
  • Baseline overlap with treatment: if users saw the new experience before formal assignment (e.g., a gradual rollout), the baseline is not clean.
  • Different baseline lengths by variant. Can happen if you add a new variant mid-experiment. Truncate all baselines to the shortest one, or restart the experiment.
  • Post-experiment dilution: including events that occurred after the experiment ended in the outcome window.

#Metric Taxonomy

Choosing the right primary metric before launch is the single most important decision in experiment design.

WHEN

Before launching any experiment. Revisit at readout if the primary metric is null but secondaries moved.

WHY

Type Definition Examples
Primary The one metric that determines ship/kill. One only. Activation rate, trial conversion, ARPA
Secondary Metrics you monitor to understand why the primary moved. Feature adoption, session depth, support tickets
Guardrail Metrics that must not degrade. Failure here blocks shipping even if primary wins. Churn rate, error rate, revenue per account
Proxy A leading indicator, measured because the true outcome takes too long. 7-day retention (proxy for 90-day LTV)

The most common mistakes: - Multiple primaries. Choosing the winner among 5 "primary" metrics after seeing results is p-hacking. One primary, full stop. - Proxy ≠ primary. If you optimize for a proxy, you may move it while the true outcome stays flat or degrades (Goodhart's Law). - Missing guardrails. Shipping a feature that improves trial conversion but increases early churn is a net loss.

HOW

  1. Write the primary metric as a formula: trial_conversions / eligible_users — not just "conversion."
  2. Power the experiment on the primary metric (see Power analysis card).
  3. List guardrails explicitly. Set the threshold at which you'd block shipping (e.g., "churn rate must not increase by more than 0.5pp").
  4. List secondaries and apply multiple testing correction at readout (see Multiple testing).
  5. Distinguish proxies: if the proxy is primary, document why you believe it predicts the true outcome.

PITFALLS

  • Ratio metrics without a denominator definition. "Conversion rate" — conversion from what? Out of all users? All visitors? All trial starters?
  • Proxies that decouple from outcomes. 7-day engagement can increase while 90-day LTV stays flat if you're attracting low-intent users.
  • Secondary metrics that are on the causal path. If the feature works by increasing engagement, then engagement is a mediator — not a secondary. Controlling for it will remove the effect.

#Power Analysis

Power analysis tells you how many users you need before you can trust a null result.

WHEN

Before launching any experiment. If you skip this, a null result is uninterpretable — you don't know whether the feature had no effect or whether you just didn't have enough data.

WHY

Statistical power (1 − β) is the probability of detecting a true effect of a given size. Standard practice: power ≥ 0.80, α = 0.05. The required sample size grows as the effect you care about shrinks or as the metric variance increases.

Four inputs: - α (false positive rate): typically 0.05 - Power (1 − β): typically 0.80 (acceptable) or 0.90 (conservative) - Baseline metric value: mean and standard deviation from historical data - MDE (minimum detectable effect): smallest effect worth acting on (see MDE card)

HOW

Binary metric (conversion rate):

n = 2 * p̄(1-p̄) * (z_α/2 + z_β)² / δ²

where p̄ = (p_ctrl + p_trt) / 2, δ = p_trt - p_ctrl.

Continuous metric (revenue, LTV):

n = 2 * σ² * (z_α/2 + z_β)² / δ²

Ratio metric: use the delta method variance (see Delta method card).

Steps: 1. Pull 14–28 days of historical data for the primary metric. Compute mean and SD from the control population. 2. Define the MDE in business terms first; convert to the metric scale. 3. Compute required n per variant. 4. Divide by your daily eligible unit count → required runtime in days. 5. If runtime > 60 days, either increase MDE, use CUPED (reduces required n by 30–50%), or reconsider the metric.

PITFALLS

  • Using inflated baselines from a previous winning experiment. The post-ship metric is higher than the steady-state baseline. Use a period with no recent experiments running on the same users.
  • Forgetting to account for clustering. If randomizing by account, multiply required n by the design effect (see Randomization unit card).
  • One-sided vs. two-sided. Use two-sided unless you have a strong prior that the effect can only go in one direction and are willing to miss a harmful effect.
  • Power for the primary only. Secondary metrics will be underpowered by definition — adjust your interpretation accordingly.

def sample_size_binary(p_base, mde_rel, alpha=0.05, power=0.80):
    p_ctrl = p_base
    p_trt  = p_base * (1 + mde_rel)
    delta  = p_trt - p_ctrl
    p_bar  = (p_ctrl + p_trt) / 2
    z_a    = norm.ppf(1 - alpha / 2)
    z_b    = norm.ppf(power)
    return int(np.ceil(2 * p_bar * (1 - p_bar) * (z_a + z_b)**2 / delta**2))

def sample_size_continuous(mean, std, mde_abs, alpha=0.05, power=0.80):
    z_a = norm.ppf(1 - alpha / 2)
    z_b = norm.ppf(power)
    return int(np.ceil(2 * std**2 * (z_a + z_b)**2 / mde_abs**2))

#MDE to Business Value

Never run an experiment that can only detect an effect too small to justify shipping.

WHEN

After computing required sample size. Translate the MDE into ARR, conversion points, or other business units before launch.

WHY

The MDE is the smallest effect your experiment can reliably detect. It is a property of your design (sample size, variance, α, power) — not a target. The question to answer before launch: "If the true effect equals the MDE, would we ship?" If the answer is no (the MDE is too small to matter), you are designing a test you'll never act on.

The reverse is also important: if the business impact of the MDE is enormous but implausible, your required sample size implies you need an unrealistically large effect — reconsider the metric or the design.

HOW

  1. Compute the MDE in the metric's native units (e.g., conversion rate percentage points, or mean revenue dollars).
  2. Multiply by the eligible user base to get aggregate impact: MDE_abs × MAU × 12 = annual revenue impact.
  3. Ask: is this impact meaningful enough that we'd act on it? If not, increase sample size until the MDE is actionable, or reconsider the primary metric.
  4. Document the MDE and its business interpretation in the experiment spec.

PITFALLS

  • Anchoring on a round MDE. "10% relative lift" feels clean but may not correspond to a meaningful business outcome.
  • Ignoring variance when translating. A 10pp lift in a 5%-base-rate metric matters more than a 10pp lift in a 90%-base-rate metric — both in absolute users and in business value.
  • Forgetting the denominator. MDE × 1,000 DAU is very different from MDE × 100,000 DAU.

#Sample Ratio Mismatch (SRM)

When your group sizes don't match the expected allocation, stop. Do not analyze results — the assignment mechanism is broken.

WHEN

Immediately after launch (check at 5% enrollment) and daily throughout the experiment.

WHY

An SRM means the randomization or logging pipeline is producing the wrong split. Even a 1% deviation can introduce systematic bias: if users who trigger some logging bug are more likely to end up in one variant, the groups are no longer exchangeable. Any downstream p-value is uninterpretable.

Common causes: - Redirect latency: users who see the treatment page load slightly slower and bounce before logging the assignment event - Bot filtering applied asymmetrically - Experiment SDK bug sending a non-uniform hash - Assignment logging fires after a user action (e.g., only logged on first click), so users who don't click are never logged

HOW

  1. At 5% enrollment, compare actual vs. expected group sizes using a chi-squared test.
  2. If p < 0.01, pause the experiment and investigate the assignment pipeline.
  3. Check: does the SRM appear immediately (assignment bug) or ramp up over time (logging latency or selective exclusion)?
  4. Do not try to "correct" for SRM by reweighting — the bias is in the data generating process, not the analysis.
χ² = Σ (observed_i - expected_i)² / expected_i

PITFALLS

  • Analyzing SRM experiments anyway. The result is not salvageable. Report the SRM and restart.
  • Checking SRM only at the end. By then you may have already peaked at results. Check early.
  • SRM within segments. The overall split can look fine while a segment (mobile users, enterprise accounts) has a severe mismatch.
  • Excluding users post-hoc to fix the ratio. This introduces selection bias worse than the original SRM.

#Intraclass Correlation (ICC) and Contamination

When users within the same cluster influence each other, standard errors are too narrow and contamination biases your effect estimate.

WHEN

Whenever the randomization unit is not the unit of analysis (e.g., randomizing by account but measuring users), or when users share a workspace, feed, or social graph.

WHY

ICC (Intraclass Correlation Coefficient) measures how similar users within the same account/cluster are. An ICC of 0.15 means 15% of outcome variance is explained by cluster membership. When you analyze at the user level but randomize at the account level, you pretend users are independent when they're not — shrinking your standard errors by a factor of √DEFF and inflating false positive rates.

Contamination is a related but distinct problem: a user assigned to control can "see" the treatment through a shared workspace, see a treated teammate's feature, or be affected by treatment-induced changes in the shared product (e.g., a new notification behavior in a shared feed). This biases your estimate toward zero (dilutes the effect).

HOW

Estimating ICC:

ICC = (MS_between - MS_within) / (MS_between + (m̄ - 1) × MS_within)

where MS_between and MS_within come from a one-way ANOVA on the metric by cluster.

Design effect:

DEFF = 1 + (m̄ - 1) × ICC

Multiply your required sample size by DEFF before launch.

Contamination check: - After the experiment, compute the fraction of control-arm users who are in an account with at least one treatment-arm user (the "contamination rate"). - If contamination rate > 10%, bias is likely — consider switching to account-level randomization.

PITFALLS

  • Ignoring ICC and reporting user-level p-values. With ICC=0.15 and 5 users/account, DEFF ≈ 1.6 — your effective n is 63% of your reported n.
  • Contamination through shared data. A/B-tested recommendation models that write to a shared user-facing table contaminate control users via the shared output.
  • Estimating ICC on the treated population. Use pre-experiment data from a clean period; ICC can shift if the treatment itself affects within-cluster similarity.

def simulate_icc(n_clusters, cluster_size, icc, treatment_effect=0.0):
    account_fx = rng.normal(0, np.sqrt(icc), n_clusters)
    user_noise  = rng.normal(0, np.sqrt(1 - icc), (n_clusters, cluster_size))
    y = (account_fx[:, None] + user_noise + treatment_effect).flatten()
    return y

def estimate_icc(y, cluster_ids):
    clusters = np.unique(cluster_ids)
    k  = len(clusters)
    ms = np.array([y[cluster_ids == c] for c in clusters], dtype=object)
    grand_mean = y.mean()
    ms_between = sum(len(m) * (m.mean() - grand_mean)**2 for m in ms) / (k - 1)
    ms_within  = sum(sum((v - m.mean())**2 for v in m) for m in ms) / (len(y) - k)

    icc_est = (ms_between - ms_within) / (ms_between + (m_bar - 1) * ms_within)
    return max(0, icc_est)

#Network Effects and Interference

When treating one user changes another user's outcome, randomizing individual users gives a biased estimate of the true policy effect.

WHEN

Whenever your product has social, marketplace, or shared-resource features: social feeds, messaging, notifications, driver/rider matching, shared workspaces, referral programs.

WHY

The standard A/B test assumes SUTVA (Stable Unit Treatment Value Assumption): the outcome of user A is not affected by the treatment assigned to user B. When users interact, this is violated. Two consequences:

  1. Diluted ITT estimate: control users benefit from treated neighbors → treatment effect appears smaller than it is.
  2. Biased total effect: the experiment misses spillover effects, which may be the dominant part of the business impact (e.g., a viral feature).

HOW

Interference type Recommended design
Social graph spillover Cluster randomization by social component (friend group, team)
Marketplace (supply affects demand) Geo-based or time-based (switchback)
Shared workspace Account-level randomization
Referral / invite flows Ego-network cluster randomization

For detecting interference post-hoc: 1. Compute the "treatment neighborhood exposure" for each control user: fraction of their contacts/teammates in treatment. 2. Regress the outcome on treatment assignment + neighborhood exposure. A significant coefficient on neighborhood exposure confirms spillover.

PITFALLS

  • Cluster randomization doesn't eliminate interference between clusters. Geographic market effects or platform-wide ranking changes can still bleed across.
  • Switchback designs require careful period length. Too short: carryover effects contaminate. Too long: you lose statistical power.
  • Estimating global treatment effect (GTE) vs. direct effect. The ITT from a cluster-randomized test estimates the GTE (including spillovers); a user-randomized test estimates only the direct effect.

#Selection Bias

The users who end up in your experiment are not a random sample of all users — and the gap matters.

WHEN

When the experiment uses a trigger-based exposure (feature must be activated to be "seen"), when you exclude certain user segments, or when you're comparing self-selected groups.

WHY

Selection bias occurs when the process that determines who enters each arm of the experiment is correlated with the outcome. In SaaS experiments this appears as:

  • Trigger bias: Only users who perform action X see the feature (e.g., clicked a settings menu). High-intent users are over-represented. The effect you measure is the effect on this selected population, not on all users.
  • Exclusion bias: You exclude bots, internal users, or new accounts — fine — but if the exclusion criteria themselves differ by variant (e.g., if the new experience makes bots look like real users), you're removing different populations from each arm.
  • Survivor bias: You analyze only users who are still active N days after assignment, excluding churned users. Churned users may differ systematically between arms. See Survivor bias card.

HOW

  1. Map the "exposure funnel": assigned → eligible → triggered → analyzed. At each stage, compute the drop-off by variant.
  2. If trigger rates differ by variant (e.g., 40% trigger in control, 50% in treatment), the triggered populations differ. Report ITT (all assigned users) as your primary estimate.
  3. For the triggered population effect (per-protocol), use IV or CATE estimation — see ITT vs per-protocol card.
  4. Verify exclusion criteria are applied symmetrically and pre-assignment.

PITFALLS

  • Reporting only on triggered users without disclosure. This is almost always upward-biased.
  • Excluding users who had an error in the treatment arm. Errors are an outcome, not a reason to exclude.
  • Post-stratification on endogenous variables (variables affected by treatment, like engagement or feature usage). These are colliders; controlling for them introduces bias.

#Novelty and Primacy Effects

Week-1 results lie. Users behave differently simply because something is new or different.

WHEN

Any time you're measuring engagement or behavioral metrics (clicks, feature usage, session length) in the first 1–2 weeks of an experiment.

WHY

Novelty effect: Treated users engage more simply because the experience is new. This inflates engagement metrics in the first week, making the treatment look better than it is in the long run.

Primacy (habituation) effect: Some users are confused or annoyed by a new UI pattern. Initial engagement drops but recovers as they adapt. This is the mirror image — treatment looks worse in week 1.

Both effects decay over time. If you stop the experiment at week 1, you measure the novelty/primacy, not the steady-state effect.

Minimum runtime recommendation: ≥ 14 days. For engagement metrics on returning users: 21 days.

HOW

  1. Plot the daily treatment effect (treatment mean − control mean) over time.
  2. If the effect declines over time → novelty. Wait until it stabilizes.
  3. If the effect increases over time → primacy / learning curve. Your early result was pessimistic.
  4. Split new users (first session after assignment) from returning users. Novelty effects are larger for returning users; new users don't know what "normal" looks like.

PITFALLS

  • Stopping when p < 0.05 in week 1. If there's a novelty effect, you'll see a false positive and ship something that doesn't improve long-run engagement.
  • Thinking novelty = fake effect. The novelty-adjusted effect (after week 1) can still be real and positive. Don't dismiss the result entirely.
  • Ignoring novelty in engagement metrics while trusting conversion metrics. Conversion metrics (trial start, purchase) are less susceptible because they happen once, not repeatedly.

#Survivor Bias in Engagement Metrics

Restricting your analysis to "active" users after assignment systematically over-selects high-quality users and inflates your results.

WHEN

Any time you're computing engagement, retention, or usage metrics on a post-assignment cohort filtered to active or retained users.

WHY

If you only analyze users who were active in week 2, you exclude users who churned in week 1. If the treatment causes some users to churn faster (or slower), the "active week 2" populations in treatment and control are not comparable — they're selected by an outcome that the treatment affected.

In SaaS, common survivor filters: - "Only users active at least once in the outcome window" - "Only accounts that reached day 7" - "Only users who triggered the feature"

Each of these creates a post-treatment filter that can differ between arms.

HOW

  1. Use ITT always. Assign every enrolled user a metric value (0 if they churned or never triggered). Do not filter.
  2. If you need to understand the effect on "engaged users," treat it as a separate segment analysis (not a filter), and note it is not causal.
  3. Check: does the fraction of users who survive to your analysis window differ by variant? If pct_survived_treatment ≠ pct_survived_control, survivor bias is present.

PITFALLS

  • Combining survivor bias with engagement metrics. "D14 engagement among users who reached D14" is almost guaranteed to be biased if treatment affects D14 retention.
  • "Active" definitions that differ by treatment. If the treatment is a new notification, "active" users may be inflated in treatment simply because notifications pull users back.
  • Interpreting survivor-biased results as causal. Any treatment effect estimate from a survivor-filtered cohort has an unknown, potentially large bias.

Metric Pitfalls

Common mistakes in how metrics are defined, aggregated, and interpreted. These apply in any phase but are most dangerous at readout.

#Simpson's Paradox

An effect that appears in the aggregate can reverse — or disappear — within every subgroup.

WHEN

When you see a positive aggregate result but have reason to believe composition differs across variants. Always check when the primary metric is a ratio and the group sizes differ.

WHY

Simpson's paradox occurs when a third variable (a confounder or composition variable) creates a weighted average that reverses the true direction of the effect within each group. In SaaS experiments, this typically happens when:

  • Variant groups differ in composition by plan tier, account size, or country
  • The metric is a ratio and the denominator differs between groups
  • A progressive rollout creates unequal time-in-experiment between groups

The paradox is not a statistical error — the aggregate number is arithmetically correct. It is an interpretation error: the aggregate conflates the treatment effect with a composition effect.

HOW

  1. Compute the metric separately within each major segment (plan, account size, country, cohort).
  2. If the direction within each segment is opposite to the aggregate, you have Simpson's paradox.
  3. The segment-level estimates are the correct causal estimates (assuming randomization was clean within segments).
  4. The aggregate can still be reported as "the effect on the full enrolled population," but the business interpretation should be the segment-level results.

PITFALLS

  • Assuming the aggregate effect is the whole story. In B2B SaaS with heterogeneous account sizes, Simpson's paradox is common because a handful of enterprise accounts dominate the aggregate.
  • Fixing Simpson's paradox by "controlling" post-hoc. If the composition difference is a consequence of the treatment (e.g., treatment caused more enterprise accounts to engage), controlling for it removes part of the causal effect.
  • Confusing Simpson's paradox with HTE. HTE (different effect sizes) is normal and interesting. Simpson's paradox (reversed effect direction) is a signal that the comparison group is wrong.

#Skewness and Heavy Tails in Revenue Metrics

A single large account can flip the sign of your experiment result. Revenue metrics require special treatment before any mean comparison.

WHEN

Any time the primary or secondary metric is revenue, ARR, LTV, or any other monetary metric.

WHY

Revenue distributions in SaaS are almost always right-skewed: most users/accounts contribute a small amount, while a few large accounts contribute most of the total. The mean of such a distribution is dominated by the tail. In a two-sample t-test, a single $200K account randomly assigned to treatment can create a statistically significant — but completely spurious — "effect."

Two practical consequences: 1. High variance → large required sample size. Power analysis must use the empirical SD, not a guess. 2. Sensitivity to outliers. The treatment-control difference in means can flip with a single observation.

HOW

  1. Plot the revenue distribution for each variant. If p99/p50 > 10, you have a heavy tail problem.
  2. Winsorize at the 99th (or 95th) percentile before computing means (see Winsorization card).
  3. Alternatively, use a log transform if the distribution is approximately log-normal.
  4. Use the Mann-Whitney U test (rank-based, robust to outliers) as a robustness check.
  5. Report both the raw mean difference and the winsorized mean difference. If they diverge, the result is driven by a small number of extreme observations.

PITFALLS

  • Reporting raw mean revenue without a tail check. Even a single reclassification or billing correction during the experiment window can dominate the result.
  • Log-transforming and interpreting as an additive effect. Log differences are multiplicative (relative) effects; convert back to dollar terms before reporting.
  • Winsorizing by variant separately. Always winsorize using the pooled distribution threshold, computed before unblinding.

#Ratio Metrics

mean(a/b) is not mean(a)/mean(b). Confusing them produces wrong point estimates and wrong standard errors.

WHEN

Any time the metric you care about is a ratio: ARPU = revenue/users, conversion rate = conversions/impressions, click-through rate = clicks/sessions, sessions per user.

WHY

Two distinct ratio estimators:

User-level ratio: compute a/b per user, then aggregate across users.

mean(revenue_i / sessions_i)          # arithmetic mean — equal user weight
n / Σ(sessions_i / revenue_i)         # harmonic mean — better for rates

Both give each user equal weight, but they answer different questions. The arithmetic mean of ratios weights every user the same regardless of denominator size. The harmonic mean of a rate answers "what rate would you observe if every user did the same number of sessions?" — it's the correct central tendency for a rate when the denominator is a resource or effort (sessions, days active, impressions). For right-skewed rate distributions, the harmonic mean is more robust to a few users with very high ratios.

Rule of thumb: if the denominator varies a lot across users (CV > 0.5), prefer the harmonic mean for user-level rate metrics.

Population-level ratio (ratio of means):

mean(revenue_i) / mean(sessions_i)  = Σrevenue / Σsessions

This is the total revenue divided by total sessions. High-volume users dominate.

The estimators are equal only when all users have the same denominator. In practice they differ, sometimes substantially.

Which to use: Use the population-level ratio when the business metric is fundamentally aggregate ("revenue per session for the product as a whole"). Use the user-level ratio when each user should have equal weight ("how does the average user's experience change?"). When choosing user-level, prefer the harmonic mean for rates with a variable denominator.

Variance of ratio metrics requires the delta method — the naive SE from dividing sample means is wrong (see Delta method card).

HOW

  1. Define explicitly: "Is this a user-level ratio or a population-level ratio?"
  2. Compute both and check whether they differ substantially. If yes, document the choice and why.
  3. Use the delta method for the standard error of a population-level ratio.
  4. For t-tests on ratios, use the linearized estimator (delta method approximation).

PITFALLS

  • Mixing user-level and population-level across variants. Both should use the same estimator.
  • Computing SE as SE(numerator)/mean(denominator). This ignores the covariance between numerator and denominator. Use the delta method.
  • Denominator = 0 for some users. If a user had zero sessions, their ratio is undefined. Exclude them from the user-level estimator (but note this changes the estimand) or use the population-level ratio.

#Aggregation Bias

The same experiment can show opposite results depending on whether you measure at the user, session, or event level.

WHEN

When a metric can be computed at multiple levels of aggregation, or when stakeholders argue about whether to use user-level or event-level metrics.

WHY

Aggregation bias occurs when the level at which you aggregate changes the relative weighting of units. High-activity users (who have many sessions or events) dominate event-level and session-level metrics but have equal weight in user-level metrics.

Example: a feature that makes heavy users slightly worse but light users much better will look: - Negative at the event level (heavy users have more events, dominate the mean) - Positive at the user level (light users are the majority, have equal weight)

Neither is "wrong" — they answer different questions. The aggregation level must match the business question.

HOW

  1. Define the question: "Is this a per-user effect or a per-interaction effect?"
  2. Compute the metric at multiple aggregation levels as a sanity check.
  3. If they disagree, investigate whether the disagreement is driven by a specific segment (heavy users, enterprise accounts).
  4. Match the aggregation level to the randomization unit.

PITFALLS

  • Session-level metrics when randomizing by user. Sessions from the same user are not independent; treating them as such inflates apparent n and compresses SE.
  • Event-level metrics for engagement features. A feature that encourages more (but lower-quality) events will look good on total event count but bad on per-session depth.
  • Switching aggregation level after seeing results. Choose the level at the time of pre-registration.

#Regression to the Mean

Top performers at baseline always look worse on re-measurement — even with no intervention. Cherry-picked cohorts are not experiment groups.

WHEN

When the experiment targets users selected on a recent extreme metric value (e.g., "users with >10 sessions last week"), or when interpreting before/after comparisons for high-performers.

WHY

Regression to the mean is a statistical artifact: any unit selected because it scored extremely on a noisy metric will, on average, score closer to the population mean on the next measurement — simply because some of their extreme score was due to random noise, not underlying quality. When you treat a high-performing cohort and observe their performance decline, the decline may be entirely regression to the mean, not an effect of the treatment.

Common SaaS scenarios: - Sending an upsell to "power users" (top 10% last month) and observing their engagement drops - Targeting "at-risk" accounts (lowest 10% last month) with a success program and concluding it worked

HOW

  1. Select a "regression to the mean check group": users who meet the selection criteria but were randomly assigned to a no-intervention control.
  2. The decline (or recovery) in the control group is the regression-to-the-mean baseline. Subtract it from the treatment group's change.
  3. Equivalently: use a proper randomized experiment with random assignment — the RTM effect cancels out between groups by design.

PITFALLS

  • Before/after comparisons on selected cohorts are almost always biased by RTM. The only way to correct for it is a concurrent control group.
  • Conflating RTM with the treatment effect. If control improves by 5% and treatment improves by 8%, the treatment effect is 3%, not 8%.
  • Using RTM as an excuse to dismiss real negative effects. If the treatment group declines more than the control group, RTM is not the explanation.

Variance Reduction

Techniques for making your experiments more sensitive — detecting real effects with fewer users or in less time.

#CUPED

Use last period's metric to remove pre-existing variation between users, cutting required sample size by 30–50%.

WHEN

When the primary metric is continuous and noisy (revenue, LTV, session count, feature usage) and you have 7–14 days of pre-experiment data for the same metric and units.

WHY

CUPED (Controlled-experiment Using Pre-Experiment Data) exploits the fact that much of the variance in a user metric is explained by that same user's historical behavior. By regressing out the pre-experiment metric (covariate), you remove variation that exists regardless of treatment — reducing residual variance and tightening confidence intervals.

The adjustment:

Y_adj = Y - θ × (X - E[X])

where: - Y is the outcome metric - X is the pre-experiment covariate (same metric, prior period) - θ = Cov(Y, X) / Var(X) is the OLS coefficient - E[X] is the population mean of the covariate

The variance reduction factor is approximately 1 - ρ²(Y, X), where ρ is the correlation. A correlation of 0.6 gives ~36% variance reduction → ~36% fewer users needed for the same power.

Critical constraint: the covariate X must be measured entirely before assignment. A covariate that even partially overlaps with the treatment period will absorb part of the treatment effect and bias the estimate downward.

HOW

  1. Pull pre-experiment metric for the same units over 7–14 days before assignment.
  2. Verify the baseline window ends before t0 (see Baseline and time zero).
  3. Compute θ using OLS on the full sample (pooled treatment + control).
  4. Compute Y_adj = Y - θ × (X - X̄).
  5. Run the t-test on Y_adj instead of Y.
  6. Confirm that E[Y_adj_treatment] - E[Y_adj_control] ≈ E[Y_treatment] - E[Y_control] — CUPED adjusts variance, not the point estimate.

PITFALLS

  • Covariate measured after assignment starts. Even a partial overlap biases the estimate. Use strict pre-assignment cutoffs.
  • Using a different metric as the covariate. CUPED works best when X is the same metric as Y (highest correlation). Using a proxy covariate reduces benefit.
  • Applying CUPED to binary metrics. The variance reduction is smaller for binary outcomes. It still works but gains are modest.
  • Units with no pre-experiment data. New users or new accounts have no history. Either exclude them from CUPED (analyze separately) or impute X with the global mean (conservative).

#Stratified Randomization and Post-Stratification

Lock in covariate balance before launch; or recapture it analytically after.

WHEN

Stratified randomization: when you have a small number of high-variance strata (plan tier, country, account size bucket) and want guaranteed balance.

Post-stratification: when you already ran the experiment but want to reduce variance using pre-experiment covariates.

WHY

In a large experiment, randomization produces approximately balanced groups by chance. In smaller experiments (< 1,000 units), unlucky imbalance on a key covariate (e.g., 60% enterprise in treatment, 40% in control) can dominate the metric difference. Pre-stratifying eliminates this risk.

Post-stratification (also called ANCOVA adjustment) is the retrospective equivalent: you include the stratum as a covariate in the analysis model, reducing the variance explained by stratum membership.

Both techniques reduce variance without introducing bias (provided the strata are pre-treatment).

HOW

Pre-stratified randomization: 1. Define strata (plan × size bucket, e.g., 4–6 strata). 2. Within each stratum, run a separate randomization with the target split. 3. At analysis, either analyze the strata separately (then aggregate) or include stratum fixed effects in a regression.

Post-stratification (ANCOVA): 1. Build a linear model: Y = α + β·treatment + Σγ_k·stratum_k + ε 2. The β coefficient is the covariate-adjusted treatment effect. 3. This is equivalent to CUPED when the strata are used as categorical covariates.

PITFALLS

  • Too many strata with small n. With 20 strata and 500 total users, some strata will have < 5 users per arm — the within-stratum estimates are unreliable.
  • Strata defined post-assignment on endogenous variables. The strata must be defined on pre-assignment covariates. If "enterprise" is defined by account size at assignment time, fine. If it's defined by feature usage in the first week, it's endogenous.
  • Ignoring stratification at analysis time. If you stratified at assignment but analyzed without stratification, you waste the variance reduction and your SEs are conservative.

#Winsorization

Cap extreme values at a principled threshold before computing means.

WHEN

Before any mean comparison on right-skewed metrics (revenue, LTV, session count, API calls). Always winsorize before computing CUPED adjustments on the same metrics.

WHY

Winsorization (capping) replaces values above a threshold with the threshold value. Unlike trimming (deletion), it preserves all observations — it just removes the distorting influence of the most extreme values on the mean.

Why not trim? Trimming changes the estimand: you're no longer estimating the effect on all users, just the non-extreme ones. If treatment causes more extreme users to appear (e.g., a viral loop drives whale usage), trimming would remove evidence of the treatment effect. Winsorization keeps those users in the sample at a capped value.

Threshold choice: 99th percentile is standard. Use 95th if the distribution is extremely heavy-tailed (e.g., API usage or storage consumption). Always compute the threshold on the pooled (treatment + control) distribution, before unblinding.

HOW

  1. Pool treatment and control metric values.
  2. Compute the 99th percentile of the pooled distribution.
  3. Replace any value above the threshold with the threshold value.
  4. Run the t-test on winsorized values.
  5. Report both the raw and winsorized estimates as a robustness check.

PITFALLS

  • Computing the threshold separately by variant. This introduces bias: if treatment has higher extreme values, its p99 is higher, and you clip more in treatment than in control.
  • Winsorizing before pooling. Same problem as above — thresholds differ by group.
  • Forgetting to winsorize the CUPED covariate. If Y is winsorized but X_pre is not, the CUPED adjustment is mismatched.
  • Winsorizing binary metrics. Winsorization has no effect on binary (0/1) variables.

#Delta Method for Ratio Metric Variance

The naive standard error for a ratio metric is wrong. The delta method gives the correct SE for any ratio of means.

WHEN

Whenever the metric is a ratio: ARPU, conversion rate (with session denominator), CTR, sessions per user, revenue per order.

WHY

If the metric is R = Σa_i / Σb_i (ratio of totals), the variance of R is not simply Var(a)/Var(b). The numerator and denominator are correlated, and that covariance matters.

The delta method (first-order Taylor expansion) gives:

Var(R) ≈ [Var(a) - 2R·Cov(a,b) + R²·Var(b)] / (n · E[b]²)

where a_i and b_i are user-level numerator and denominator, and expectations are taken over users.

In practice, this can be implemented as a linearized OLS: regress the "linearized" outcome a_i - R·b_i and use its SE.

HOW

  1. Compute R = mean(a) / mean(b) — the population-level ratio.
  2. Compute linearized outcome: z_i = a_i - R · b_i
  3. Use the standard t-test or OLS on z_i — its SE is the delta method SE for R.
  4. Divide the t-test result by mean(b) to get back to the ratio scale.

PITFALLS

  • Computing SE as SE(a)/mean(b). Ignores the covariance term and is usually too narrow (optimistic).
  • Forgetting to linearize before CUPED. If you're combining CUPED with a ratio metric, linearize first, then apply CUPED on the linearized outcome.
  • Using ratio metrics with very small denominators. When b_i is often 0 or 1, the delta approximation breaks down. Use the user-level ratio (mean(a_i/b_i)) instead.

#Propensity Score Matching (PSM)

Match treated and control units on their probability of receiving treatment, then compare outcomes within matched pairs.

WHEN

Observational analysis: a feature was shipped to some users without randomization (e.g., a self-serve activation flow, early-adopter cohort). You want to estimate its effect on revenue or retention.

WHY

When treatment is not randomly assigned, treated and control users differ systematically (confounding). Propensity Score Matching creates a pseudo-experimental comparison by matching each treated user to a control user with similar observable characteristics — similar "propensity" to be treated.

The propensity score e(X) = P(T=1 | X) is estimated by logistic regression on observed covariates. After matching, the matched control group is a valid comparator conditional on the observed covariates — but unmeasured confounders can still bias the result.

Key assumption: Conditional exchangeability — there are no unmeasured confounders. In observational SaaS data, this is strong. Always argue for it explicitly.

HOW

  1. Identify observed confounders: variables that predict both treatment adoption and the outcome (account age, plan, usage level, industry).
  2. Fit a logistic regression: P(T=1) ~ X1 + X2 + ... + Xk.
  3. For each treated user, find the nearest control user by propensity score (1:1 nearest-neighbor matching, with or without replacement).
  4. Assess balance: compare covariate means in matched treatment vs. control. Standardized mean differences (SMD) should be < 0.1.
  5. Estimate the average treatment effect on the treated (ATT) from the matched sample.

PITFALLS

  • Matching on many covariates without checking overlap. In high-dimensional spaces, some treated users may have no close control matches — these units are discarded, changing the estimand.
  • Reporting ATT as if it were ATE. PSM estimates the effect for users who were treated. If high-intent users self-selected in, the ATT overestimates the effect on the average user.
  • Not checking balance after matching. The matching algorithm may fail silently if the propensity distributions don't overlap.
  • Ignoring time: if treatment adoption happened gradually, match within time windows to avoid comparing users from different periods.

#Difference-in-Differences (DiD)

Compare the change over time in a treated group against the change in a concurrent control group.

WHEN

When you have a geo-based or time-based rollout (a feature launched in some markets or for some account cohorts) and you have pre-rollout data for both the treated and untreated groups.

WHY

DiD controls for time-invariant confounders (e.g., enterprise accounts are always higher-revenue) by differencing them out. The estimator:

DiD = (Ȳ_treated_post − Ȳ_treated_pre) − (Ȳ_control_post − Ȳ_control_pre)

The treated group's change captures both the time trend and the treatment effect. Subtracting the control group's change removes the time trend — leaving only the treatment effect.

Critical assumption: parallel trends. Before the rollout, the treated and control groups must have been trending in the same direction. If enterprise accounts were already growing faster than SMBs before the feature launched, the DiD estimate will be wrong.

HOW

  1. Define the treated group (got the rollout) and control group (didn't).
  2. Pull the primary metric for both groups for a pre-period (≥2 weeks) and a post-period.
  3. Verify parallel trends: plot the metric for both groups in the pre-period. They should move together.
  4. Estimate the DiD using OLS: Y = α + β₁·treated + β₂·post + β₃·(treated × post) + ε β₃ is the DiD estimate.
  5. Add covariates to absorb remaining variance (DiD + regression adjustment).

PITFALLS

  • Parallel trends violated. The most common failure. If treated accounts were growing faster before rollout, DiD overestimates the effect. Test with a placebo pre-period DiD.
  • Rollout happened gradually. "Event study" designs (staggered DiD) are needed when treatment dates differ across units.
  • Wrong control group. Using a control group that was exposed to a different change at the same time (e.g., got a different feature) makes the trends non-parallel.
  • Post-period too short. If the feature takes time to show effects, a short post-period will underestimate.

#Instrumental Variables (IV)

Use a variable that causes treatment but has no direct effect on the outcome to isolate the causal effect of treatment.

WHEN

When treatment is not randomly assigned, but there exists a "nudge" variable (the instrument) that shifts who receives treatment without directly affecting the outcome.

WHY

In many SaaS products, randomization is used as an encouragement design: users are randomly shown (or not shown) a prompt to activate a feature. The prompt is the instrument Z; whether they actually activate is the treatment T; the outcome is Y.

The IV estimator recovers the Local Average Treatment Effect (LATE): the causal effect for users who complied with the instrument (activated because they were prompted — "compliers"). This is not the ATE for all users.

The three IV conditions (Hernán & Robins): 1. Relevance: Z is associated with T (prompt increases activation). 2. Exclusion restriction: Z affects Y only through T (the prompt itself doesn't change revenue — only activation does). 3. Independence: Z is as good as randomly assigned (the prompt was A/B tested).

LATE = ITT_Y / ITT_T = (effect of prompt on Y) / (effect of prompt on T)

HOW

  1. Verify the instrument is randomly assigned (check for SRM, balance).
  2. First stage: estimate the effect of Z on T (compliance rate). If < 10%, the instrument is weak.
  3. Reduced form: estimate the effect of Z on Y (ITT_Y).
  4. IV estimate: LATE = ITT_Y / ITT_T (Wald estimator for binary Z and T).
  5. For continuous or multiple instruments, use 2SLS.

PITFALLS

  • Weak instruments (low first-stage F < 10). Leads to severely biased and wide IV estimates. Report the first-stage F-statistic.
  • Exclusion restriction is untestable. You cannot test whether the instrument affects Y directly. Build the strongest possible argument from domain knowledge.
  • LATE ≠ ATE. Compliers may be a small, unrepresentative subpopulation. The effect on non-compliers is not estimated.
  • Using IV when ITT is sufficient. For most product decisions, the ITT (intent-to-treat) estimate is the right answer. IV is needed when you specifically want the per-protocol effect.

#Doubly Robust Estimation (AIPW)

Combine an outcome model with a propensity model — so that the estimate is consistent if either one is correctly specified.

WHEN

Observational analysis where you want protection against misspecification of either the propensity score or the outcome model — the best default for causal estimation from observational SaaS data.

WHY

Augmented Inverse Probability Weighting (AIPW) combines: 1. A propensity model e(X) = P(T=1|X) — models who gets treated 2. An outcome model μ(X, T) = E[Y|X, T] — models how outcomes relate to covariates

The doubly robust estimator is consistent if either model is correctly specified. It is the most robust observational estimator for average treatment effects.

The AIPW estimator:

τ̂_AIPW = E[μ(X,1) - μ(X,0)]
        + E[(T/e(X)) · (Y - μ(X,1))]
        - E[((1-T)/(1-e(X))) · (Y - μ(X,0))]

In practice, use the DoubleML library or implement it manually.

HOW

  1. Split the data into two halves (cross-fitting to avoid overfitting).
  2. On the first half, fit the propensity model e(X) and outcome model μ(X, T).
  3. On the second half, compute the AIPW score for each observation.
  4. Average the scores to get the ATE estimate. Swap halves and repeat.

PITFALLS

  • Extreme propensity scores. If some units have e(X) ≈ 0 or 1, the IPW component explodes. Trim propensity scores to [0.05, 0.95].
  • Using the same data for model fitting and estimation (no cross-fitting). This introduces regularization bias. Always use cross-fitting.
  • Misinterpreting "doubly robust" as "always right." If both models are misspecified, the estimate is wrong. Doubly robust protects against one being wrong.

    # Outcome model

    mu1 = out_model.predict(np.column_stack([X_ev, np.ones(len(X_ev))]))
    mu0 = out_model.predict(np.column_stack([X_ev, np.zeros(len(X_ev))]))

    # AIPW score
    aipw_scores[eval_idx] = (mu1 - mu0
        + T_ev / e_hat * (Y_ev - mu1)
        - (1 - T_ev) / (1 - e_hat) * (Y_ev - mu0))

ate_aipw = aipw_scores.mean()
se_aipw  = aipw_scores.std() / np.sqrt(n)
naive    = Y[T==1].mean() - Y[T==0].mean()

ps_full = LogisticRegression().fit(X, T)
e_full  = np.clip(ps_full.predict_proba(X)[:, 1], 0.05, 0.95)
ipw_ate  = (Y * T / e_full - Y * (1 - T) / (1 - e_full)).mean()

#Target Trial Emulation

Design your observational analysis as if you were running an RCT you wish you had run.

WHEN

When you have observational log data (feature adoption logs, usage data) and want to answer a causal question that you didn't run a formal experiment for.

WHY

Target trial emulation (Hernán & Robins, Ch. 22) is a framework that forces you to be explicit about the hypothetical RCT you're trying to emulate. By specifying the components of that target trial, you define the estimand precisely and identify what assumptions are needed to answer the causal question with observational data.

The key components of a target trial: 1. Eligibility criteria: Who would have been enrolled? 2. Treatment strategies: What are the two (or more) interventions being compared? 3. Assignment mechanism: How would units be randomly assigned? 4. Follow-up: When does follow-up begin and end? 5. Outcome: What is the primary endpoint? 6. Causal contrast: ATE, ATT, or per-protocol effect?

In SaaS, this is most useful for estimating the causal effect of product decisions that were shipped without a randomized test.

HOW

  1. Write out the target trial spec in plain language.
  2. Map each trial component to observational data:
  3. Eligibility → SQL filter
  4. Treatment → activation event, plan change, or feature first-use
  5. Time zero → date of activation or enrollment
  6. Follow-up → outcome window after time zero
  7. Apply PSM, DiD, or AIPW to estimate the treatment effect under the stated assumptions.
  8. Conduct sensitivity analyses: what would an unmeasured confounder of a given magnitude need to look like to explain away the result?

PITFALLS

  • Immortal time bias. If time zero is set to feature first-use, users who never use the feature are excluded. Set time zero to the date they became eligible, not the date they used it.
  • Prevalent user bias. Enrolling users who have already been using the product for varying durations means users differ in stage-of-product-relationship at enrollment. Restrict to new users or new accounts for cleaner emulation.
  • Not stating the assumptions. Target trial emulation is not a way to avoid stating causal assumptions — it's a way to make them explicit so they can be evaluated.

#Heterogeneous Treatment Effects (HTE)

The average effect hides the distribution of effects. Some segments gain, others lose.

WHEN

After seeing the aggregate result — always, as a secondary analysis. Required when the aggregate effect is null (could be averaging out opposite effects in subgroups).

WHY

The average treatment effect (ATE) is a population average. In SaaS, the effect of a feature almost always differs across account size, plan tier, usage level, geography, or user role. A feature that is neutral on average could be strongly positive for SMBs and strongly negative for Enterprise — shipping it harms a segment.

HTE analysis can be: 1. Pre-specified subgroup analysis: defined before unblinding; valid for causal claims within subgroups (still requires multiple testing correction). 2. Exploratory CATE estimation: discover which variables moderate the effect; hypothesis-generating only.

The most common CATE methods: - Stratification: split by segment, compute within-segment ATE - Interaction model: Y ~ treatment * segment + covariates - Causal forests (GRF): non-parametric CATE for continuous heterogeneity

HOW

  1. List the 3–5 segments to test, pre-specified before unblinding.
  2. Compute the ATE within each segment.
  3. Test for effect modification: Y ~ treatment + segment + treatment:segment. The interaction coefficient's p-value tests whether the effect differs across segments.
  4. Correct for multiple comparisons (Bonferroni if segments are few; Benjamini-Hochberg for exploratory).
  5. Present as a forest plot: effect and CI per segment.

PITFALLS

  • Post-hoc segment fishing. Testing 20 segments after seeing the aggregate result inflates false positives. Pre-register the segments.
  • Segment n is too small. A within-segment test requires the same power as the overall test — and smaller n. Report CIs, not just point estimates.
  • Confusing HTE with Simpson's paradox. HTE is about effect size varying across segments. Simpson's is about effect direction reversing due to composition differences. Both can be present simultaneously.
  • Segment membership is post-treatment. Only use segments defined on pre-treatment covariates.

#Multiple Testing Correction

Testing N metrics at α = 0.05 produces ~0.05N false positives in expectation. Pre-specify corrections before unblinding.

WHEN

Whenever you report more than one hypothesis test: multiple secondary metrics, multiple segments, multiple time windows.

WHY

Each test at α = 0.05 has a 5% chance of a false positive. Testing 20 metrics independently gives an expected 1 false positive — and an actual false positive rate much higher than 5% for the full set (family-wise error rate, FWER).

Two common correction approaches:

Bonferroni: multiply each p-value by N (or equivalently, use α/N as the threshold). Controls FWER. Very conservative for large N — many true effects become non-significant.

Benjamini-Hochberg (BH): controls the False Discovery Rate (FDR) — the expected proportion of significant results that are false positives. Less conservative than Bonferroni; appropriate when you can tolerate some false positives among secondary findings.

Practical rule: Apply no correction to the single pre-registered primary metric. Apply Bonferroni or BH to secondary metrics.

HOW

Benjamini-Hochberg procedure: 1. Sort p-values from smallest to largest: p₁ ≤ p₂ ≤ ... ≤ pₘ 2. For each p_k, compute the BH threshold: q_k = α × k / m 3. Find the largest k where p_k ≤ q_k 4. Reject all hypotheses with p-values ≤ p_k

PITFALLS

  • Applying Bonferroni to 50 metrics. The required α becomes 0.001 — most real effects won't reach significance. Use BH for large metric sets.
  • Testing segments as if they were independent hypotheses. Segment tests are correlated (all come from the same experiment). Report them as exploratory, not primary.
  • Correcting the primary metric. The primary metric is the one pre-registered hypothesis. No correction needed.
  • Reporting "marginally significant" secondary metrics. p = 0.06 after BH is not significant. Do not massage the language.

#ITT vs. Per-Protocol Analysis

ITT measures the effect of being assigned to treatment. Per-protocol measures the effect of actually receiving it. Both are valid answers to different questions.

WHEN

When a non-trivial fraction of users assigned to treatment never experienced it (e.g., they never triggered the feature, or you're comparing plans but some users didn't upgrade).

WHY

Intent-to-Treat (ITT): Analyze all assigned users regardless of whether they actually received the treatment. This is the correct estimate for "what happens if we ship this feature to all eligible users?" It includes users who would never activate the feature.

Per-Protocol (PP): Analyze only users who complied with their assignment (activated the feature if in treatment, didn't use it if in control). This estimates "what is the effect for users who actually use the feature?" PP is biased by self-selection unless handled via IV (see IV card).

Typical pattern: ITT is always the primary estimate. PP is reported as secondary, with explicit IV correction for self-selection bias.

HOW

  1. Report ITT as primary: include all assigned users, 0 outcome for non-activators.
  2. Compute compliance rate: fraction of treatment-assigned users who activated.
  3. If PP is requested, use the IV/Wald estimator: PP estimate = ITT_Y / compliance_rate.
  4. Report both with clear labeling of the estimand.

PITFALLS

  • Reporting only PP without disclosure. PP estimates are systematically higher than ITT because only high-intent users comply. Presenting PP as "the effect" overstates impact.
  • Compliance rate < 20%. At low compliance, ITT is heavily diluted. This is a design problem (the feature is hard to find or low-value) rather than an analysis problem.
  • Defining "compliance" on the control side. Control-side contamination (control users who somehow get the treatment) should be accounted for in the IV framework, not silently excluded.

#Mediation

Is the feature working through the mechanism you designed — or through a different path?

WHEN

When the primary metric moved but you want to understand why. Also when the primary metric didn't move but a secondary did, and you want to know if the secondary is a mediator or a confounder.

WHY

A mediator M is a variable that sits on the causal path from treatment T to outcome Y. The total treatment effect decomposes into: - Direct effect (CDE/NDE): T → Y, not through M - Indirect effect (NIE): T → M → Y

Example: a new onboarding flow (T) improves 90-day revenue (Y). The mechanism might be: better onboarding → higher feature adoption (M) → higher revenue. The indirect effect through M tells you whether to invest in more onboarding or in the feature itself.

Hernán & Robins define mediation precisely using potential outcomes and require four assumptions: (1) no unmeasured T→Y confounders, (2) no unmeasured M→Y confounders, (3) no unmeasured T→M confounders, (4) no M→Y confounder affected by T.

HOW

For a linear model (Baron-Kenny, appropriate for continuous M and Y): 1. Fit: M ~ α + βT·T 2. Fit: Y ~ γ + βT'·T + βM·M 3. Indirect effect: βT·βM 4. Direct effect: βT' 5. Total: βT' + βT·βM 6. Proportion mediated: βT·βM / total

For non-linear models, use causal mediation formulas or the causalml / mediation packages.

PITFALLS

  • Conditioning on a post-treatment mediator that is itself confounded. If M→Y is confounded by a variable affected by T, the decomposition is biased. Use the cross-world potential outcomes approach.
  • Interpreting mediation results causally when assumptions aren't met. Mediation analysis requires strong untestable assumptions. Label results as "exploratory."
  • Using mediation to explain a null. "The effect was null because M didn't move" is a causal claim that requires the same assumptions.

#Sequential Analysis and Early Stopping

Run a valid hypothesis test at any point during the experiment without inflating the false positive rate.

WHEN

When you need to make a decision before the pre-specified sample size is reached — due to business urgency, a harmful treatment arm, or unexpected rapid enrollment.

WHY

Peeking at p-values repeatedly and stopping when p < 0.05 inflates the false positive rate to ~47% with weekly peeks over 5 weeks. Sequential methods (SPRT, always-valid inference) provide valid tests at any stopping time by pre-specifying the test boundaries.

SPRT (Sequential Probability Ratio Test): Define boundaries A and B before launch. At each peek, compute the likelihood ratio. Stop if it crosses A (reject null) or B (accept null).

Always-valid inference / e-values: A newer approach that maintains validity at any stopping time with no pre-specified boundary. More flexible for continuous monitoring.

HOW

SPRT for proportion metrics: 1. Pre-specify H₀: p = p₀ and H₁: p = p₁ (the MDE lift from p₀). 2. Set α (false positive rate) and β (false negative rate). 3. Compute boundaries: A = (1-β)/α, B = β/(1-α). 4. After each new observation, update the log-likelihood ratio. 5. If log LR ≥ log A → reject H₀. If log LR ≤ log B → accept H₀. Otherwise continue.

PITFALLS

  • Using fixed-sample p-values with sequential stopping. This is the most common mistake — stopping when a regular t-test shows p < 0.05 while peeking repeatedly.
  • Forgetting to set boundaries before launch. The SPRT boundaries cannot be adjusted after peeking.
  • Confusing sequential stopping with bayesian updating. SPRT is frequentist. For a Bayesian equivalent, see Bayesian readout.
  • Applying SPRT to revenue metrics without transformation. SPRT assumes a specific distribution family. For complex metrics, use the mixture-sequential test or always-valid inference.

#Bayesian Readout

Express results as "probability of being better" and "expected loss" instead of p-values and confidence intervals.

WHEN

When communicating to non-technical stakeholders, when you want to incorporate a prior, or when the "probability of beating control" is more decision-relevant than p < 0.05.

WHY

The frequentist p-value answers: "If there were no true effect, how surprising is this data?" This is rarely what decision makers want to know.

Bayesian inference answers: "Given this data, what is the probability that the treatment is better than control?" This is directly interpretable and actionable.

For conversion rate metrics, the conjugate Bayesian A/B test uses Beta distributions: - Prior: Beta(α₀, β₀) — uninformative: Beta(1, 1) (uniform) - After observing s successes in n trials: posterior Beta(α₀ + s, β₀ + n - s) - P(treatment > control) = integral of the posterior region where μ_trt > μ_ctrl (computed via Monte Carlo or the regularized incomplete beta function)

For continuous metrics, use normal conjugate models or sample from a t-distribution posterior.

HOW

  1. Define the prior. Use an uninformative prior (Beta(1,1)) unless you have strong historical evidence.
  2. Update the posterior with the observed data.
  3. Compute P(treatment > control) via Monte Carlo sampling.
  4. Compute expected loss: E[max(0, μ_ctrl − μ_trt)] — the cost of shipping if you're wrong.
  5. Report the posterior distribution, the probability, and the expected loss.

PITFALLS

  • Using an informative prior to rescue a failing experiment. If the prior is constructed from the current data, you're circular.
  • Reporting P(treatment > control) = 0.8 as "80% chance this works." It is the probability the effect is positive, not the probability the product decision is correct.
  • Ignoring the magnitude. A 95% probability that treatment is better by $0.01 is not the same as a 95% probability it's better by $10.

#Translating CIs into Business Language

"We are 95% confident the effect is between X and Y" is not the same as "there's a 5% chance this is wrong." Translate carefully.

WHEN

When writing the experiment readout document, preparing a stakeholder summary, or presenting results in a business review.

WHY

Confidence intervals and p-values are frequently misinterpreted — even by experienced analysts. The correct interpretation of a 95% CI is:

"If we ran this experiment many times, 95% of the intervals constructed this way would contain the true effect."

What it is NOT: - "There is a 95% probability the true effect is in this interval" (that's a Bayesian credible interval) - "The true effect is almost certainly in this range" (it's either in there or it isn't)

For business communication, translate the CI into business units and focus on the decision boundaries: - The lower bound of the CI is the pessimistic scenario — the worst plausible outcome if you ship. - The upper bound is the optimistic scenario. - If the lower bound is positive and above the minimum meaningful effect, ship with confidence. - If the lower bound is negative, name the downside risk explicitly.

HOW

  1. Convert the CI endpoints to business units: revenue, conversion rate percentage points, accounts, ARR.
  2. Frame as a decision boundary: "Even in the pessimistic scenario (lower bound = X), we gain $Y ARR."
  3. State the probability language only for Bayesian results. For frequentist CIs, use "plausible range" or "confidence interval."
  4. Always report effect size alongside significance. A statistically significant p-value with a tiny effect size is not a business win.

PITFALLS

  • Reporting p-values without effect sizes. "p = 0.03" tells a business stakeholder nothing about magnitude.
  • Treating the CI midpoint as "the" effect. The point estimate is just the most likely value — the full interval is the uncertainty.
  • "Highly significant" without context. With N = 1,000,000, nearly any effect is statistically significant. Significance is not importance.
  • Mixing frequentist and Bayesian language. "95% probability the effect is positive" (Bayesian) and "p < 0.05 two-sided" (frequentist) are different claims. Don't use both in the same sentence.

#When to Say "Caused" vs. "Associated With"

The word "caused" requires three things: a clean randomization, no SRM, and no obvious alternative explanation.

WHEN

When writing the conclusion of an experiment readout or presenting results to leadership.

WHY

"Caused" is a strong claim. It requires that the causal identification conditions are met (see Foundations): 1. Exchangeability: groups were comparable before treatment (clean RCT, passed SRM check, passed balance check) 2. Positivity: all units had a chance of being in either group 3. Consistency: the treatment was well-defined and applied uniformly 4. No alternative explanation: novelty effects ruled out (≥ 14-day experiment), no parallel business changes, no contamination

When all four are satisfied, you can write: "The new checkout flow caused a 2.1pp increase in trial conversion rate."

When one or more are violated or uncertain, use: "The new checkout flow was associated with a 2.1pp increase in trial conversion rate among users who encountered it."

For observational analyses (PSM, DiD, IV), the claim is: "After adjusting for observed confounders, the feature was associated with..."

HOW

Pre-readout causal checklist: - [ ] Randomization verified (SRM check passed, balance table clean) - [ ] Runtime ≥ 14 days (novelty effects decayed) - [ ] No concurrent business changes affecting the metric - [ ] No contamination detected (ICC check, trigger rate symmetric) - [ ] Exclusion criteria applied symmetrically - [ ] Primary metric pre-registered (no p-hacking)

If all boxes are checked → causal language is warranted. If any box is unchecked → qualify the claim.

PITFALLS

  • Using "impact" as a hedge. "The feature impacted conversion" is ambiguous — it sounds causal but means nothing precise. Use "was associated with" or "caused."
  • Observational analyses claiming causation. PSM and DiD are not RCTs. Label them "quasi-experimental" and state the assumptions.
  • Causal language in secondary metrics. "Treatment caused higher engagement" — only if engagement was pre-registered as a causal outcome, not a secondary observation.

#Writing the Experiment Narrative

A good experiment readout fits on one page and drives a clear decision. It has a structure, not just a list of numbers.

WHEN

When writing any experiment readout — for a weekly metrics review, a launch decision, or a post-mortem.

WHY

Stakeholders read experiment readouts to make decisions, not to learn statistics. A readout that leads with p-values and data quality notes will lose most readers before the conclusion. The narrative should lead with the decision-relevant finding and back it up with the evidence.

HOW

One-page structure:

## [Feature Name] Experiment — [SHIP / KILL / ITERATE / INCONCLUSIVE]

**Hypothesis:** If we [change], we expect [metric] to [change] because [mechanism].

**Setup:**
- Primary metric: [metric]  MDE: [X]  Power: [80%]  Runtime: [14 days]
- Randomization: [unit]  Sample: [n] per variant

**Result (primary metric):**
- Effect: [+X pp / +$Y / +Z%]
- 95% CI: [lower, upper]
- p-value: [X]  (Bayesian P(better): [Y%])
- ARR impact (at lower bound): [$X]

**Secondary metrics:**
| Metric | Direction | Significant? | Concern? |
|--------|-----------|--------------|---------|
| [metric] | [↑↓→] | [Y/N] | [Y/N] |

**Guardrails:** All passed / [ALERT: churn rate increased by X]

**Conclusion:** [One sentence: what happened and why it matters]

**Decision:** Ship / Kill / Iterate — [reason]

**Caveats:** [Novelty effect possible; HTE on Enterprise not powered]

PITFALLS

  • Leading with data quality. Move SRM, balance, and quality checks to an appendix. They matter, but they're not the story.
  • Listing all metrics without priority. The primary metric is the headline. Everything else is context.
  • "Inconclusive" without a recommendation. An inconclusive result is still a decision — usually "don't ship." Say so.
  • No caveats section. Every experiment has limitations. Name them. Omitting them makes the result look falsely clean.

#Communicating Null Results

A null result is a valid, useful finding. Communicate it as a decision, not a failure.

WHEN

When the experiment ends with no statistically significant effect on the primary metric.

WHY

A properly powered null result provides strong evidence that the true effect is smaller than the MDE. This is valuable information: it means the feature doesn't move the metric at the scale detectable with the current design, and the team should either iterate on the feature, re-examine the hypothesis, or deprioritize the metric.

Null results are routinely under-reported, which leads to: - Re-running the same failed features without acknowledging prior evidence - Publication bias in internal experiment records - Loss of learning — the hypothesis was wrong, which is worth knowing

A null result is a business decision: "we have enough evidence that this feature doesn't move conversion rate by more than the MDE to justify not shipping it."

HOW

  1. Confirm the experiment was adequately powered: actual n ≥ required n.
  2. Report the effect estimate and CI: "The effect was +0.2pp (95% CI: [-0.8pp, +1.2pp]). We can rule out effects larger than +1.2pp at 95% confidence."
  3. State the MDE and whether it was achieved: "Our MDE was ±0.8pp. The upper bound (+1.2pp) is above the MDE — the result is inconclusive, not a clean null."
  4. Distinguish true null (CI entirely below MDE) from underpowered (CI straddles MDE or is wide).
  5. Recommend: iterate on the hypothesis, increase the effect size, or sunset the feature.

PITFALLS

  • Treating null as "nothing happened." The null says "we couldn't detect an effect of the size we cared about." That's different from "there is no effect."
  • Reporting null without checking power. A null from an underpowered experiment is uninformative.
  • "The experiment failed." The experiment succeeded — it gave you evidence. The feature's hypothesis was wrong.
  • Using null to "prove the null hypothesis." Absence of evidence is not evidence of absence, especially when n is small.

            MAX(CASE WHEN variant='treatment' THEN std*std/n END) +
            MAX(CASE WHEN variant='control'   THEN std*std/n END)
        )                                                   AS se,
        MAX(n)                                              AS n_per_variant
    FROM results
)
SELECT
    effect,
    effect - 1.96 * se AS ci_lower,
    effect + 1.96 * se AS ci_upper,
    se,
    n_per_variant,
    CASE
        WHEN ABS(effect) + 1.96 * se < 0.008  -- MDE = 0.8pp
        THEN 'True null: CI entirely below MDE'
        WHEN ABS(effect) < 0.008 AND effect + 1.96 * se > 0.008
        THEN 'Inconclusive: CI straddles MDE — may be underpowered'
        ELSE 'Effect detected or CI above MDE'
    END AS verdict
FROM comparison
